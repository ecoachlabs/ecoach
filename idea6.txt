Thought for 8s
A knowledge gap is the specific part of what a student does not yet know, understand, retain, or apply correctly.

In your kind of educational system, it should not be treated as just “the student is weak in math.” That is too broad.

It should be broken into layers:

1. Content gap
The student has not learned the concept at all.
Example: they do not know what a fraction numerator means.

2. Understanding gap
They have seen it before, but do not truly understand it.
Example: they can recite a formula but cannot explain why it works.

3. Application gap
They understand the concept in theory, but cannot use it in questions.
Example: they know percentages but fail word problems.

4. Process gap
They know the idea, but their steps are weak.
Example: in algebra they make the wrong first move.

5. Speed gap
They can solve it, but too slowly for exam conditions.

6. Accuracy gap
They know it, but make careless mistakes.

7. Retention gap
They once knew it, but forgot it.

8. Confidence gap
They hesitate, panic, or avoid the topic, so performance drops even when some knowledge exists.

9. Interest gap
They are disengaged, so they never spend enough quality effort to improve.

10. Transfer gap
They can solve familiar questions but fail when the question is asked in a new way.

For your app, a knowledge gap should be modeled as:

Topic → Subtopic → Skill → Error pattern → Cause → Fix path

Example:

Subject: Mathematics

Topic: Fractions

16. Gap shrinking logic
The total gap should shrink only when real evidence appears.

Do not shrink because the student watched a lesson.

Shrink only when the student demonstrates:

correctness

repeat correctness

transfer ability

acceptable speed

later retention

So one successful question should reduce the gap a little, not fully.

A skill becomes truly reduced when there is stable evidence.

Example repair levels:

explanation viewed: no major gap reduction

guided correct: slight reduction

independent correct: medium reduction

repeated correct across variation: major reduction

retained later: long-term reduction

This prevents fake mastery.

17. Solidification strategy types
Different skills need different repair styles.

For unknown skills
Teach from scratch.

For weak skills
Target specific breakdown points.

For declining skills
Reinforce quickly and retest.

For forgetting skills
Use spaced retrieval.

For misconception-heavy skills
Confront the wrong mental model directly.

For speed gaps
Use timed repetition after accuracy stabilizes.

So the app should not repair everything the same way.

11. State transition rules
Each skill should follow a controlled state machine.

Here is a clean version.

unknown → weak
When first evidence shows low mastery.

unknown → fragile
When first evidence is mixed but incomplete.

weak → repairing
When a repair session starts.

repairing → fragile
When partial improvement appears but not enough proof yet.

fragile → stable
When the student performs consistently across varied items.

stable → mastered
When performance is sustained across time and retention checks.

mastered → at_risk_mastered
When decay or decline signals appear.

at_risk_mastered → declining
When performance drops below threshold.

declining → critical
When trend worsens or downstream failures appear.

critical → repairing
When intervention starts.

repairing → stable
When sufficient evidence proves recovery.

This state machine should be explicit in code.

12. Hidden blocker detection logic
This is one of the smartest parts of the system.

A hidden blocker means:
the student appears weak in one area, but the real cause is elsewhere.

Example:
student keeps failing ratio questions, but root cause is equivalent fractions.

Detection approach
A. observe downstream failures
Student is repeatedly failing children skills

B. inspect prerequisite graph
Check upstream skills linked to those failures

C. compare performance
If upstream skill has weak/fragile evidence and children are failing, flag probable blocker

D. compute blocker confidence
A blocker confidence score can be derived from:

dependency strength

upstream weakness severity

downstream failure frequency

recency

E. mark blocker
Set:

is_hidden_blocker = true

create feed item

increase priority

This is how the app says:
“You seem weak in ratios, but your real issue is fractions.”

13. Forgetting / decay model
This mode must detect not just ignorance, but slipping knowledge.

A practical first model:

Each skill has:

memory_strength

last_correct_at

predicted_decay_at

next_retention_check_at

Basic logic
If a skill has not been reinforced for a while, decay risk rises.

Example
mastery high + recently reinforced = low forgetting risk

mastery medium + long inactivity = medium risk

previously mastered + recent errors = high risk

Simple decay approach
Use a decay curve where memory strength gradually falls with time unless reinforced.

Every independent successful recall:

increases memory strength

pushes predicted_decay_at further out

Every failed recall:

reduces memory strength sharply

raises forgetting risk

can move skill to at_risk_mastered or declining

This is how the app protects what the student already learned.

25. Final backend definition
The backend for Knowledge Gap Mode is a student-centered academic state engine built around per-skill knowledge records, event-driven evidence capture, live gap and priority scoring, dependency-aware blocker detection, and intervention planning. 26. The most important design rule
Do not let the system confuse activity with mastery.

Watching content is activity.
Tapping through screens is activity.
Reading an explanation is activity.

The gap should only meaningfully shrink when there is evidence of:

correctness

repeat correctness

variation success

retention

usable speed

reduced misconception

That one rule will protect the integrity of the whole feature.
