11. Question assembly logic
After ranking, the engine must still assemble properly.

This is important because top-ranked questions alone can create a bad session:

too much of one topic

poor pacing

punishing openings

no emotional rhythm

Assembly constraints
do not overrepeat one micro-skill unless intended

maintain planned topic balance

maintain difficulty curve

avoid too many same-shaped questions in a row

ensure at least one confidence-stabilizing moment if needed

ensure coverage of key selected areas

ensure trap count does not exceed plan

Ordering patterns
Pattern A — Warm / Core / Trap / Stabilize
Best for class tests and confidence modes.

Pattern B — Sectioned Simulation
Best for mocks and structured exams.

Pattern C — Diagnostic / Repair / Confirm
Best for weakness repair and teach-through-test.

Pattern D — Speed Ladder
Best for pressure modes.

The blueprint should choose one.

Adaptation policy levels
None
No changes once session starts.
Best for full simulation.

Light
Only insert internal notes and maybe change next item difficulty slightly.

Moderate
Can reorder upcoming items and insert repair checks.

High
Can pause, teach, simplify, and retest.

Adaptation triggers
Trigger 1 — Same misconception streak
If the student makes 3 errors with the same misconception tag,
then:

flag misconception as active

insert a targeted repair item

optionally show a short concept correction

Trigger 2 — Slow but accurate
If accuracy is high but response time exceeds target repeatedly,

mark speed weakness

introduce a time-compressed familiar item later

Trigger 3 — Fast but sloppy
If speed is high but careless errors spike,
then:

introduce precision prompt

Trigger 4 — Confidence collapse
Detected by:

repeated skips

rapid guessing

long freezes

performance crash after hard items

Then:

insert stabilizer question

preserve progression

Trigger 5 — Overperformance
If performance is too easy:

27. Risks and design cautions
Risk 1 — Weak question tagging
If your question bank is poorly tagged, this feature becomes shallow.

Risk 2 — Overpromising predictions
Do not imply certainty when generating “likely questions.”
It should be high-probability preparation, not guaranteed prediction.

Risk 3 — Too much adaptation in simulation mode
If simulation keeps helping, it stops being simulation.

Risk 4 — Demotivating low-confidence students
A harsh sequence right before a test can reduce performance rather than improve it.

Risk 5 — Generic results
If the result page only shows score, the feature loses its premium value.

If days_left 2–5
Set:

urgency = near

If days_left 6–14
Set:

If days_left > 14
Set:

Urgency changes everything downstream.

If one topic or tightly related topics
Set:

If 2–4 topics with moderate spread
Set:

If many topics or whole unit
Set:

Weak

Unstable

9. Adaptation logic in detail
9.1 Same misconception streak logic
If:

2 or 3 wrong answers in same misconception family

adaptation policy >= moderate

Then:

mark misconception as active

promote targeted recheck or corrective question into next 2 slots

optionally display a short correction card

Example:
Student keeps confusing:

respiration

photosynthesis

System inserts:

a clean comparison item

then a short application item

9.2 Confidence stabilization logic
Signals:

long pause

rapid guess after hard question

repeated wrong answers

optional self-reported low confidence

Then:

insert confidence-stabilizer question

avoid immediate stacking of another harsh trap

keep student moving

This matters especially near a real test.

9.3 Speed weakness logic
If:

correct answers but too slow

relevant in timing-heavy mode

Then:

mark speed weakness

include familiar but quicker item later

result page highlights speed risk separately from knowledge risk

9.4 Overperformance logic
If:

accuracy very high

time very good

difficulty too easy relative to profile

Then:

increase upcoming item complexity

or insert one stretch item

especially in mock or elite style prep

13. Engineering notes
A few important implementation choices:

13.1 Store blueprint snapshot
Once session starts, store the blueprint as a snapshot.
Do not recompute it from scratch later unless adaptation explicitly changes it.

13.2 Adaptation should be event-driven
Do not continuously rerun huge logic.
Trigger checks after each answer.

13.3 Separate question ranking from question assembly
Ranking picks good candidates.
Assembly creates a good session.
These should not be mixed.

13.4 Keep simulation mode strict
If simulation mode adapts too much, it stops being a simulation.

13.5 Scope profile should be cached
Student scoped performance for common topic groups should be cached where possible.
