3. How the test should be structured
The test should happen in stages.

Stage 1: Fast baseline scan
Goal: quickly map broad strengths and weaknesses.

The software asks a spread of questions across the whole subject:

Stage 2: Topic zoom
After the baseline scan, the software automatically zooms into weak or unstable areas.

If the student struggles in:

the engine branches deeper into those specific areas.

Stage 3: Misconception probing
This is critical.

The software should not only ask for the right answer.
It should ask in ways that expose why the student is wrong.

Example in math:
If a student keeps adding denominators in fractions, that is not just “wrong.”
That is a specific misconception.

Example in biology:
If the student confuses mitosis and meiosis, the software should determine whether the confusion is about:

genetic variation

So each wrong option in the database should be tagged to a likely misconception.

That way, at the end, the report can say:

Student is not simply weak in fractions. Student specifically treats numerators and denominators as independently addable units.

That is much more useful than “43% in fractions.”

Stage 4: Speed and pressure layer
A student may know a topic in calm mode but fail in exam mode.

So for selected items, the software should test in two conditions:

untimed or relaxed timing

tighter timing / pressure timing

This reveals:

knows under no pressure

breaks under pressure

slow but accurate

fast but careless

panic-driven guessing

This distinction matters enormously.

Because the intervention for “does not know” is different from the intervention for “knows but collapses under time.”

Stage 5: Transfer layer
The student answers one form of the question, then the engine tests the same concept in a different form.

Example:

word problem version

diagram version

comparison version

explain-why version

This stage reveals whether the student has real understanding.

Stage 6: Confidence capture

This is powerful because it tells you:

confidently wrong

confused and aware

“Confidently wrong” is one of the most important things to detect.

Later in the same session, reintroduce a few concepts indirectly.

inconsistent

variation

Time: 8–12 minutes
Questions: 6–10

Time: 5–8 minutes
Questions: 4–8

Purpose:

Method:
Bring back earlier concepts indirectly through sibling items.

Not the same question.
A related question.

Later:

unstable mastery

Time: 1–2 minutes

Ask short reflective items like:

Which parts felt confusing?

This should not override system findings, but it adds useful contrast between:

actual evidence

6. How to detect guessing precisely
The engine should not classify a correct answer as “known” immediately.

Each correct answer gets an evidence quality rating.

prerequisite consistency

recall vs recognition comparison

A likely guessed answer may look like:

low confidence

long hesitation

multiple option changes

later sibling wrong

related prerequisite weak

A securely known answer may look like:
correct

moderate/high confidence

appropriate speed

later sibling also correct

recall item also correct

So each correct answer falls into:

likely guessed

This is how you solve the “what they guessed” problem.

9. Recognition vs recall test design
This is critical.

5. How the system actually generates questions
Now we get to the real generation engine.

The best architecture is not one method.
It is three methods working together.

Method 1: Template-based generation
This is the backbone.

You define structured templates like:

“Which of the following best describes {concept}?”

“What is the main difference between {termA} and {termB}?”

“In the diagram below, which label represents {feature}?”

“If {scenario}, what will happen to {object}?”

“A student says {incorrect_statement}. Why is this wrong?”

Templates are good because they are:

controlled

explainable

consistent

easy to validate

good for offline systems

Method 2: Parameterized generation
This is especially important for mathematics and some sciences.

Here the question has a stable structure, but variables change.

Example:

numbers change

objects change

names change

units change

order changes

contexts change

Math example
Template:
“Simplify: {a}/{b} + {c}/{d}”

Constraints:

denominators may be same or different

answers must reduce nicely

distractors must reflect real misconceptions

Physics example
“A body moves at {speed} for {time}. Find the distance.”

Grammar example
“Choose the sentence with the correct use of {tense/form}.”

Parameterized generation is how you create depth without repeating exact items.

Method 3: Controlled AI-assisted generation
This should be used carefully.

AI can help:

rewrite stems

change style

generate scenario variants

produce alternate phrasings

generate plausible distractors

create sibling items

But it must operate inside constraints.

AI should not be allowed to freely invent the academic truth from scratch.

It should work from:

known concept atoms

known answer key

known misconceptions

known difficulty target

known format template

So AI becomes a variation assistant, not the source of truth.

That is the safest model.

5. Rules for a Root Concept Problem
This is one of the most important types.

A root concept problem exists when a foundational atom is broken and that break explains several downstream issues.

Detection conditions
Flag root concept problem if:

core atom mastery < 35

rootness score > 0.70

downstream impact breadth > 0.60

at least 2 affected downstream subskills/topics show weakness

evidence confidence ≥ 0.75

Rootness score inputs
prerequisite centrality

number of dependent atoms

frequency of downstream failure correlation

whether fixing this concept would plausibly improve multiple areas

Example
Weak denominator meaning
→ fraction comparison weak
→ unlike denominator operations weak
→ ratio interpretation weak

This is not just a fraction problem.
It is a root concept problem.

6. Rules for a Misconception Problem
This exists when the same wrong idea appears repeatedly and consistently.

Detection conditions
Flag misconception problem if:

same misconception appears in at least 3 relevant items

misconception confidence > 0.70

wrong-answer signature is interpretable

error appears across more than one wording or form if possible

wrong responses are not purely random guesses

Strong misconception case
Especially strong if:

wrong answers are high-confidence

rechecks preserve the same wrong logic

pressure worsens the same misconception

Example
“Adds numerators and denominators directly”
If detected in 4–5 items, that becomes a real misconception problem card.

7. Rules for a Translation Problem
This is common in mathematics, science applications, and language-heavy tasks.

A translation problem means the student often knows the concept in direct form but struggles to convert language, situation, or representation into the right method.

Detection conditions
Flag translation problem if:

direct-form score ≥ 60

applied/verbal/word-problem score ≤ 40

transfer drop ≥ 15

recall/recognition is not enough to explain the whole gap

repeated hesitation or wrong operation selection in language-based items

Strong translation case
Especially strong if:

operation choice follows keywords instead of structure

equation formation is weak while direct solving is okay

diagrams/scenarios break performance while direct form survives

This is extremely important because many parents think the problem is “math,” when it is actually translation.

8. Rules for a Condition Problem
This means the student knows more in one condition than another.

Typical cases:

calm vs timed

recognition vs recall

first pass vs recheck

direct vs transfer

Detection conditions
Flag condition problem if there is a large performance gap across conditions.

Timed pressure problem
calm accuracy – timed accuracy ≥ 16

and timed evidence count sufficient

and calm condition is meaningfully stronger

Recognition dependence
recognition – recall gap ≥ 18

and recall evidence count sufficient

Transfer weakness
direct – transfer gap ≥ 15

and direct accuracy sufficiently higher

Stability weakness
first-pass correct to recheck retention < 60%

with enough recheck samples

If condition gaps are large, the problem card should be about the condition, not just the topic.

9. Rules for a Reliability Problem
This is where correctness is misleading.

A reliability problem exists when the student gets items right, but the correctness is fragile, guessed, or unstable.

Detection conditions
Flag reliability problem if:

guess likelihood in the area > 0.25

or fragile correct proportion > 0.20

or stability recheck failure high

or correct answers rely heavily on recognition only

or answer-change pattern suggests uncertainty

Strong reliability case
Especially strong if:

topic raw score looks decent

but transfer, recall, or recheck is weak

so the area appears stronger than it is

This creates powerful cards like:
“Fractions look stronger than they are.”

10. Rules for a Fluency Problem
This exists when the student mostly understands but is too slow or inefficient.

Detection conditions
Flag fluency problem if:

mastery ≥ 60

but speed index ≤ 45

and timed performance falls

while misconception density is not the main driver

and accuracy in calm condition remains decent

This distinguishes:

“doesn’t know”
from

“knows but is not fluent”

That matters a lot for intervention.

11. Rules for a Memory / Retention Problem
This is when the issue is not first-pass understanding, but keeping it stable over time or across rechecks.

Detection conditions
Flag retention problem if:

first-pass correctness reasonably good

but recheck or delayed sibling performance drops sharply

stability index < 45

transfer not necessarily the main issue

misconception not dominant

This kind of problem says:

understood briefly

did not consolidate

12. Rules for a Discrimination / Confusion Problem
This occurs when similar concepts are getting mixed up.

Examples:

osmosis vs diffusion

mitosis vs meiosis

ratio part-to-part vs part-to-whole

area vs perimeter

Detection conditions
Flag confusion problem if:

errors cluster specifically around one confusion pair

contrast items fail more than isolated items

student picks features belonging to the wrong paired concept

misconception map links to a confusable concept

This should generate cards like:
“The student is confusing X with Y, not failing both equally.”

That is a very different diagnosis.

13. Candidate problem formation rules
Once all weak patterns are detected, the system creates problem candidates.

A candidate should be formed when:

one broken root atom is found
or

one active misconception is strongly repeated
or

one condition gap is large and meaningful
or

one translation pattern is repeated
or

one reliability distortion is large enough to matter

Each candidate then gets typed and scored.

14. Candidate merging rules
You do not want 15 overlapping problem cards.

The engine should merge candidates that describe the same underlying issue.

Merge if:
same root atom cluster

same misconception drives them

same downstream chain

same fix path

high semantic overlap

Do not merge if:
causes differ

fixes differ

one is root and the other is a distinct condition problem

they matter independently

Example
These may merge:

fraction comparison weak

unlike denominator alignment weak

These may remain separate:

denominator meaning weak

timing collapse in word problems

Because one is conceptual root, the other is a condition problem.

This keeps the report clean.

confidence too low

6. Repair rule for root concept problems
When the engine detects a root concept problem, the repair plan should almost always follow this order:

Stage 1 — Rebuild meaning
The student must understand what the atom actually means.

Stage 2 — Contrast against confusions
Separate it from similar ideas.

Stage 3 — Apply in low-pressure direct form
Simple direct use

Stage 4 — Apply in varied form
Different contexts and phrasings

Stage 5 — Recheck stability
Use sibling items

Stage 6 — Add timing only after stability
Never too early

Rule
If root concept broken:
meaning before procedure
contrast before speed
stability before pressure

This should be a core repair principle.

7. Repair rule for misconception problems
A misconception cannot be fixed by random repetition.

It requires explicit correction.

Standard misconception repair sequence
Step 1 — Surface the wrong idea
The system must make the student confront the faulty logic.

Step 2 — Explain why it feels tempting
This is important psychologically.

Step 3 — Contrast wrong vs right
Side-by-side

Step 4 — Rebuild the correct rule or meaning
Not just “don’t do that”

Step 5 — Immediate controlled practice
Small, focused items targeting the same trap

Step 6 — Recheck with a different surface form
Make sure the misconception truly weakened

Rule
If misconception active:
expose → contrast → rebuild → test

Not:
repeat → repeat → repeat

8. Repair rule for translation problems
Translation weakness means the student often knows the concept but cannot map language or representation into the correct method.

Standard translation repair sequence
Step 1 — Extract quantities / entities
What is given?

Step 2 — Extract relationships
How do the parts connect?

Step 3 — State what is being asked
What is the target?

Step 4 — Map relationship to operation / concept
This is the key repair stage

Step 5 — Solve in scaffolded form
Structured solving

Step 6 — Fade the scaffold
Reduce help

Step 7 — Reintroduce novel contexts
Transfer

Rule
If translation problem:
structure before solving

Do not jump straight to doing more full word problems.

9. Repair rule for recall-vs-recognition problems
If the student can recognize but not recall, then MCQ-heavy practice may actually hide the weakness.

Standard recall repair sequence
Step 1 — Remove options
Use low-stakes recall

Step 2 — Cue lightly, not fully
Partial prompts only

Step 3 — Require self-generation
Definition, method, label, step, equation

Step 4 — Check correctness
Immediate correction

Step 5 — Repeat under spaced retrieval
Not mass repetition only

Step 6 — Recheck in another form
Make sure the recall became stable

Rule
If recognition > recall:
generate before selecting

That should be built into the library.

10. Repair rule for pressure problems
If the student knows more in calm condition than under timing, then timing should be introduced carefully.

Standard pressure repair sequence
Step 1 — Stabilize the concept in calm condition
No point timing broken knowledge

Step 2 — Build efficient retrieval pattern
Short guided bursts

Step 3 — Use soft timing
Light urgency, not harsh pressure

Step 4 — Use repeated timed mirrors on already repaired items
Only after stage 1–2 succeed

Step 5 — Increase time pressure gradually
Not suddenly

Step 6 — Recheck whether accuracy stays intact
If accuracy collapses, drop back

Rule
If pressure-sensitive:
accuracy first, then speed

Never the other way around.

11. Repair rule for transfer weakness
Transfer weakness means the student knows the familiar form but not the idea deeply enough.

Standard transfer repair sequence
Step 1 — Reconfirm direct understanding
Make sure core knowledge exists

Step 2 — Show the same concept in two different wrappers
Example:
definition form and scenario form

Step 3 — Ask what stays the same across forms
This is critical

Step 4 — Practice across multiple wrappers
diagram, words, example, application

Step 5 — Test in a new wrapper
No hint that it is the same concept

Rule
If transfer weak:
invariance before novelty

Meaning:
teach the student what remains constant across changed surfaces.

12. Repair rule for stability problems
If first-pass correctness is not holding, the problem is consolidation.

Standard stability repair sequence
Step 1 — Re-teach the fragile atom briefly
Do not assume first-pass success means mastery

Step 2 — Immediate recheck
Same session

Step 3 — Short delayed recheck
Later in sequence

Step 4 — Alternate-form recheck
Sibling item

Step 5 — Spaced retrieval cycle
Come back after interval

Rule
If stability weak:
verify persistence, not just immediate success

13. Repair rule for fluency problems
If understanding is decent but response is slow, then repair must target efficiency without sacrificing correctness.

Standard fluency repair sequence
Step 1 — Confirm concept secure
No fluency work on broken concepts

Step 2 — Reduce cognitive load
Chunk the method

Step 3 — Repeated accurate short bursts
Speed windows

Step 4 — Immediate feedback
Prevent fast bad habits

Step 5 — Gradually compress response time
Only as accuracy stays high

Rule
If fluency weak:
compress after security

13. How multiple dives work in one session
A strong DNA session will usually contain multiple diagnostic dives.

Example:

one in Fractions

one in Word Problems

one in Ratio

one in Algebra under timing

These should not all happen back-to-back in a chaotic way.

The orchestrator should manage them.

Rule of thumb
DNA should not stay in a dive forever.

It should:

go deep enough to isolate

confirm enough to trust

then move on

This prevents the session from feeling stuck.

14. The best orchestration principle
The best overall session design is:

Broad scan with real-time interrupt dives
That means:

DNA is always trying to maintain broad subject coverage

but it is allowed to interrupt itself whenever a meaningful weakness emerges

This is better than:

doing all broad scan first, then diagnosis later
because your whole idea is that diagnosis happens live

And it is better than:

diving on every wrong answer
because that becomes noisy

So the correct model is:
broad flow, interrupted by intelligent descents

16. The exact stopping rules for a dive
DNA needs clear rules for when to stop digging.

A dive should end when one of these happens:

A. Cause isolated with high confidence
DNA now knows the real issue.

B. Cause isolated with moderate confidence and time discipline requires exit
Good enough for session constraints.

C. Weakness not confirmed
The signal looked suspicious, but the deeper probe did not support it.

D. Enough evidence collected to create a problem card
The dive has done its job.

E. Student recovery has been confirmed
The system can climb.

Without stopping rules, DNA will over-dig.

This is how DNA earns the right to say:

These findings should not rest on a single moment.
