9. Core scoring logic
Accuracy Rate

Pace

Daily Growth Delta
This compares today to yesterday.

Volume delta

Correct delta
correct_delta = today_correct - yesterday_correct

Pace delta
pace_delta = yesterday_avg_pace - today_avg_pace

Positive pace delta means faster today.

18. Feedback component behavior
Feedback must be carefully designed.

Correct answer feedback
Should be short and energizing:

Correct

Nice

Good pace

Strong answer

Wrong answer feedback
Should be informative but light:

Almost

Watch the denominator

Sign error here

Read the keyword carefully

Avoid harsh red failure states that demoralize.

Advanced option
The feedback can change based on mode:

on speed days: quicker feedback

on accuracy days: slightly richer feedback

on recovery mode: gentler tone

19. Timer behavior
The timer is a powerful emotional tool.
Use it carefully.

Timer styles by mode

prominent timer

Recovery Mode
relaxed timer or extended time

UX principle
Do not make the interface constantly scream time pressure.
Pressure must be controlled.

20. Motion and animation principles

24. Empty states
You need strong empty states.

First-time user empty state
No yesterday data exists.

The app should show:

baseline quick assessment

message: “Let’s build your starting point.”

Then after first session:
today becomes tomorrow’s baseline.

No progress data yet
Show:
“Complete a few climbs to see your trend.”

No badges yet
Show:
“Your first badge is one session away.”

25. Error and interruption states
Network interruption
If online:

save progress locally

show non-panicking message

let student continue if possible

Accidental exit
Prompt:
“Leave today’s climb?”
Options:

Continue

Exit session

Session timeout
If session expires or app closes:

allow resume if policy permits

Question load failure
Show:
“Couldn’t load the next question. Try again.”

The emotional tone must stay calm.

26. Accessibility and usability rules
This is very important because the audience includes weak or stressed students.

Rules
large readable text

strong contrast

clear tap/click zones

no tiny timers

consistent button locations

keyboard support on desktop

screen reader-friendly structure where possible

color not being the only indicator of performance state

If the student is mentally strained, usability matters even more.

33. Personalization layer in frontend
The frontend should eventually personalize tone and layout emphasis slightly.

Example:

some students respond more to streaks

some respond more to graphs

some respond more to simple “beat yesterday” cards

some respond better to calmer layouts

So eventually the UI can emphasize different cards based on engagement behavior, while keeping core structure the same.

39. What makes this frontend successful
A strong frontend for this feature does 5 things well:

1.
Makes today’s target obvious

2.
Makes the session easy to enter

3.
Makes the question flow clean and focused

4.
Makes improvement visible immediately after session

5.
Makes long-term rise visible over time

If any of these fail, the system loses power.

40. Final product summary
Beat Yesterday’s frontend is not just a question interface.

It is a daily climb interface.

It must help students feel:

guided

challenged

safe

measured

improving

The strongest core loop is:

see target → take session → see growth → return tomorrow

That loop is the product.
