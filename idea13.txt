5. The core indexes
Now let’s define the engine metrics.

I’ll use family-level notation because that is where most of the real value sits. You can also compute many of these at question-pattern level.

Let:

P = total number of papers in the selected window

F = a question family

A(F) = set of papers in which family F appears

count(F) = total number of question instances belonging to F

Q_p = total questions in paper p

5.1 Exact Repeat Count
For a canonical pattern C:

ERC(C) = number of instances linked to C by exact/template repeat

This answers:
How many times did essentially this same question return?

Visible output:

exact repeats

normalized repeats

years of repeat

5.2 Family Recurrence Rate
For family F:

FRR(F) = |A(F)| / P

This answers:
In what proportion of papers does this family appear?

If F appears in 8 out of 10 papers, FRR = 0.8.

Visible output:

Appears in 80% of papers in this window

This is one of your strongest metrics.

5.3 Family Density
FD(F) = count(F) / sum(total questions across all papers)

This answers:
How much of the exam landscape is occupied by this family?

A family may appear often but only once per paper. Another may appear less often but in multiple forms.

This gives better texture.

5.4 Yearly Persistence Score
For each family over a timeline:

YPS(F) = longest continuous streak of years in which F appears / total years in window

This answers:
Does this family keep coming steadily, or is it sporadic?

5.5 Dormancy Score
DS(F) = maximum consecutive years absent

This answers:
How long can this family disappear before returning?

This matters because some families go dormant, then re-enter in mutated form.

5.6 Co-Appearance Strength
For families F1 and F2:

Basic form:
CAS(F1,F2) = |A(F1) ∩ A(F2)| / |A(F1) ∪ A(F2)|

That is Jaccard similarity.

But Jaccard alone is not enough. You also want directional lift.

Lift(F2|F1) = P(F2 appears | F1 appears) / P(F2 appears overall)

This answers:
When F1 appears, does F2 become more likely than usual?

That gives stronger intelligence.

Visible output:

Often travels with

strongly associated with

weak association

rare companion

5.7 Inverse Appearance Index
This is your differentiator.

You want to know which families tend to disappear when another appears.

For families F1 and F2:

Let:

ObservedJoint = P(F1 and F2 appear together)

ExpectedJoint = P(F1 appears) * P(F2 appears)

Then:

IAI(F1,F2) = max(0, (ExpectedJoint - ObservedJoint) / ExpectedJoint)

Interpretation:

near 0: no inverse behavior

closer to 1: strong suppression or mutual exclusion

This is elegant because it compares observed joint appearance against what we would expect if they were independent.

You can also compute a directional variant:

DIAI(F2 suppressed by F1) = 1 - P(F2 appears | F1 appears) / P(F2 appears overall)

If this is high, F2 becomes much less likely when F1 is present.

Visible output:

tends to displace

often suppresses

mutually exclusive with

rarely seen together

This is the feature that makes people stare.

5.8 Replacement Index
You asked not just what vanishes, but what takes its place.

For families F1 and F2, where F2 may be a replacement candidate for F1:

RI(F1→F2) = weighted combination of

inverse strength between F1 and F2

chronological handoff pattern

structural similarity

cognitive similarity

role similarity inside paper

A practical version:

RI(F1→F2) = 0.35*IAI(F1,F2) + 0.25*ChronoShift + 0.20*StructureSimilarity + 0.20*IntentSimilarity

Where:

ChronoShift checks whether F1 declines over recent years as F2 rises

StructureSimilarity checks if they occupy similar curriculum territory

IntentSimilarity checks if they test similar skills or paper roles

This tells you:
“This family is not just absent. It is being replaced by this other family.”

That is premium intelligence.

5.9 Mutation Drift Score
Within a family, questions change.

You want to know how far a current variant has moved from the family’s older center.

For a question instance q in family F:

MDS(q,F) = weighted distance from family centroid

Distance can use:

wording complexity

context type

number of solution steps

trap count

abstraction level

mark value

reasoning depth

This answers:
Is this an old-school direct version or a modern evolved version?

Visible output:

high mutation

modern evolved variant

5.10 Cognitive Demand Vector
Each question gets a vector like:

Each value can be 0–5 or 0–1.

This allows powerful similarity clustering.

Two questions can now be linked not because topic is same, but because thinking pattern is same.

5. Similarity and grouping logic
To compute families, you need a layered similarity model.

5.1 Exact similarity
Used for direct repeats.

Signals:

normalized text equality

same numeric values

same options

same answer

same structure

5.2 Template similarity
Used for question forms that differ only in variable names, nouns, or cosmetic context.

Signals:

variable normalization

entity replacement

same solution path

same answer schema

5.3 Structural similarity
Used for different wording but same underlying reasoning skeleton.

Signals:

same solution operation chain

same logical dependency steps

same concept dependency tree

same response format

5.4 Family similarity
Used for broader clustering.

Signals:

same topic/subtopic

similar cognitive vector

similar misconception vector

similar examiner intent

related structure class

The system should not collapse these layers into one. Each layer tells a different truth.

6. Core index formulas
Now the intelligence metrics.

Assume:

P = all papers in selected window

F = family

A(F) = papers where F appears

C(F) = total question instances belonging to F

Q_total = total number of question instances across all papers in the window

6.1 Exact Repeat Count
For canonical pattern C:

ERC(C) = number_of_question_instances_matching_C_at_exact_or_template_level

Display:

exact repeats

near repeats

repeat years

6.2 Family Recurrence Rate
FRR(F) = |A(F)| / |P|

This is the share of papers in which the family appears.

Interpretation:

0.80 means the family appears in 80% of papers

6.3 Family Density
FD(F) = C(F) / Q_total

This shows how much space the family occupies across the paper universe, not just whether it appears.

6.4 Persistence Score
If the time window is ordered by year:

PS(F) = longest_consecutive_presence_streak / total_years

This tells you whether the family is stable or patchy.

6.5 Dormancy Score
DS(F) = longest_consecutive_absence_streak

This becomes interesting for dormant-returning families.

6.6 Co-Appearance Strength
For families F1 and F2:

CAS(F1,F2) = |A(F1) ∩ A(F2)| / |A(F1) ∪ A(F2)|

This is the Jaccard-style view.

But the more useful signal is directional lift:

Lift(F2|F1) = P(F2 present | F1 present) / P(F2 present)

Interpretation:

1 means F2 becomes more likely when F1 appears

< 1 means F2 becomes less likely when F1 appears

6.7 Inverse Appearance Index
This is your signature metric.

For F1 and F2:

Let:

p1 = P(F1 present)

p2 = P(F2 present)

p12 = P(F1 and F2 present together)

Expected co-appearance under independence:
E = p1 * p2

Then:

IAI(F1,F2) = max(0, (E - p12) / E )

Interpretation:

0 means no inverse behavior

closer to 1 means strong suppression

Directional version:

DIAI(F2 suppressed by F1) = max(0, 1 - P(F2 present | F1 present) / P(F2 present))

This tells you whether F2 becomes less likely when F1 is present.

This is the exact metric behind your “inverse appearance index” concept.

6.8 Replacement Index
You do not just want suppression. You want substitution.

For families F_old and F_new:

RI(F_old → F_new) = 0.30*IAI + 0.25*TrendHandoff + 0.20*TopicOverlap + 0.15*CognitiveOverlap + 0.10*PaperRoleOverlap

Where:

IAI measures inverse relation

TrendHandoff measures old family decline while new family rises over recent years

TopicOverlap checks whether they live in similar curricular space

CognitiveOverlap checks whether they test similar mental moves

PaperRoleOverlap checks whether they occupy similar places in the paper

Interpretation:
High RI means the newer family looks like a successor or substitute.

6.9 Mutation Drift Score
For question q inside family F:

MDS(q,F) = weighted_distance(q_vector, family_centroid_vector)

Where the vector may include:

wording complexity

contextualization level

number of reasoning steps

trap density

abstraction level

mark value

cognitive depth

Interpretation:

low = classical family form

medium = moderate mutation

high = modern evolved variant

6.10 Cognitive Fingerprint Vector
Each question should receive a vector like:

CFV(q) = [recall, interpret, translate, compute, justify, multi_step, eliminate, pressure]

Each score may be 0 to 1 or 0 to 5.

This vector is essential for clustering beyond topic boundaries.

6.11 Misconception Exposure Score
For question q and misconception m:

MES(q,m) = weighted_probability_that_failure_on_q_is_explained_by_m

Signals:

distractor patterns

wrong working steps

latency behavior

explanation text

partial-mark structure

family history

6.12 Trap Density Score
TDS(q) = weighted_sum(trap_features)

Trap features:

distractor plausibility

hidden unit shift

irrelevant detail burden

wording ambiguity risk

order-of-operations bait

familiar-but-wrong shortcut temptation

overconfidence bait

6.13 Story Role Score
SRS(q,p) = weighted_combination(position, relative_difficulty, discrimination, time_burden, trap_density, novelty)

Then classify each question into role buckets such as:

opener

stabilizer

separator

speed trap

confidence breaker

capstone

7. Family construction logic
Question families cannot be created by topic only. That would be shallow.

A family should be built from a blended similarity rule.

7.1 Family clustering features
Use:

subtopic overlap

solution-structure similarity

cognitive similarity

misconception similarity

examiner-intent similarity

answer-schema similarity

context-type proximity

7.2 Family admission rule
A question enters a family if:

it meets minimum topical alignment

and reaches threshold on either structure or cognitive similarity

and does not violate exclusion rules

Example:
A graph-based linear question and a pure sentence-based linear question may still live in the same family if they test the same translation and method-selection skeleton.

7.3 Family exclusion rules
Questions should not be merged just because they share topic.

Example:
“simple substitution” and “symbolic translation from word problem” may both be algebra, but may belong to different families if the cognitive and trap profiles differ significantly.

9. Data pipeline
This is the operational flow.

Step 1: ingest paper
Input may be scanned paper, PDF, manually entered text, or structured archive data.

Step 2: segment paper
Split into sections, questions, subparts, options, marking scheme items.

Step 3: normalize question
Clean OCR noise, standardize symbols, normalize punctuation, abstract variable placeholders, extract numerical structure.

Step 4: map curriculum tags
Topic, subtopic, micro-skill, objective.

Step 5: compute cognitive profile
Estimate mental operations demanded.

Step 6: compute misconception profile
Estimate what errors are likely exposed.

Step 7: pattern matching
Try exact match, template match, structural match, then family match.

Step 8: assign or update family
Attach to existing family or create candidate family for review.

Step 9: compute graph relationships
Update co-appearance, inverse, replacement, evolution, and similarity edges.

Step 10: update stories
Refresh question stories, family stories, paper stories, trend stories.

Step 11: overlay student data
When student responses exist, update student-specific interpretations.

10. Relationship inference rules
These rules stop the graph from becoming noisy.

10.1 Exact repeat rule
Create EXACT_REPEAT_OF only when normalized template and answer schema both match above a strict threshold.

10.2 Structural similarity rule
Create STRUCTURALLY_SIMILAR_TO when solution dependency tree similarity crosses threshold, even if wording differs strongly.

10.3 Co-appearance rule
Only create CO_APPEARS_WITH after minimum paper-support threshold is met. Do not create from one or two coincidences.

10.4 Inverse rule
Only create INVERSE_TO if:

IAI exceeds threshold

support count is sufficient

no false inverse due to low-frequency sparsity

10.5 Replacement rule
Only create REPLACES when:

inverse behavior exists

timeline handoff exists

topical/cognitive overlap is meaningful

10.6 Evolution rule
Create EVOLVES_INTO when newer variants preserve family logic but differ in contextualization, reasoning depth, or presentation mode.

Part 4: Interaction philosophy
The brilliance of the system should be revealed progressively.

Do not dump every metric on every screen.

Use a three-depth interaction model.

Depth 1: Quick insight
Every screen should offer a quick takeaway.

Example:
“This family appears in 78% of papers and has become more contextualized.”

Depth 2: Hidden structure
When the user expands, show:

co-appearance

inverse

replacement

mutation

family relations

Depth 3: Expert detail
Only deeper expansion reveals:

exact metrics

timeline evidence

misconception distributions

student overlays

story role logic

This keeps the UI elegant while preserving depth.

premium

investigative

alive

That way the visual system teaches meaning.
