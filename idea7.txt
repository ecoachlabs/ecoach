Below is the full product spec for Memory Mode.

Memory Mode — Full Product Specification
1. What Memory Mode is
Memory Mode is a focused learning mode that tracks what a student has already learned, measures how strongly it still exists in memory, detects when that memory is starting to fade, and immediately launches the right recovery path before the weakness spreads.

This mode is not about teaching a brand-new concept from scratch. It is about protecting, rebuilding, and strengthening what has already entered the student’s mind.

The system treats memory as a living structure, not a static score. A concept can be:

strong,

shaky,

fading,

or broken.

And because concepts are connected, a weakness in one small subtopic can reduce performance in neighboring areas. Memory Mode exists to catch that early.

2. Core promise
The promise to the student is:

“We help you hold on to what you’ve learned, catch forgetting early, and rebuild weak memory before it damages your results.”

The promise to the product is:

Memory Mode should reduce silent forgetting, improve recall speed, stabilize performance over time, and strengthen cross-topic retention.

3. What counts as “memory” in the system
Inside the product, memory is not a single thing. It has six measurable dimensions.

A. Retention
Can the learner still answer correctly after time has passed?

B. Recall speed
How quickly can they pull the answer from memory?

C. Recall stability
Do they remember it consistently across multiple attempts?

D. Transfer recall
Can they still answer when the question changes form?

E. Connection strength
Can they use the concept inside related or combined questions?

F. Recovery ability
When the concept weakens, how quickly can the learner restore it?

These six dimensions form the internal definition of memory.

4. Knowledge model behind the scenes
For Memory Mode to work well, every question must be tagged properly.

Each question should be linked to:

subject

topic

subtopic

micro-skill

prerequisite skills

neighboring/related skills

question family

question variant type

cognitive action required

misconception type

expected response time

difficulty level

memory cue type

Example
A question may be tagged like this:

Subject: Mathematics

Topic: Fractions

Subtopic: Equivalent Fractions

Micro-skill: Simplify fraction to lowest terms

Prerequisite: common factors

Related skills: ratio simplification, percentage reduction

Variant type: multiple choice

Misconception: dividing numerator only

Expected time: 18 seconds

This is critical because Memory Mode does not intervene at the broad topic level first. It intervenes at the smallest meaningful recall unit.

7. Main scoring model
The core metric should be the Memory Stability Index (MSI), scored from 0 to 100.

Components
Let:

A = recent accuracy score

S = recall speed score

R = retention-over-time score

V = variant transfer score

I = independence score

C = connection score

Then:

MSI = (0.30 × A) + (0.15 × S) + (0.20 × R) + (0.15 × V) + (0.10 × I) + (0.10 × C)

How each component works
A. Accuracy score
Weighted toward recent attempts.

Example:

last 5 attempts weighted 60%

previous 10 attempts weighted 40%

B. Speed score
Compare learner response time to expected response time.

Example:

if expected is 20 sec and learner averages 18 sec, speed is strong

if learner averages 39 sec, memory may exist but retrieval is weak

A simple normalized version:

S = clamp(expected_time / actual_time, 0, 1)

C. Retention score
Measures whether the concept survives time gaps.

The learner may get it right immediately after practice but fail after 4 days.
That means true retention is weak.

D. Variant transfer score
Measures whether the learner recalls the concept when the presentation changes.

Example:

definition form

application form

reverse form

mixed context form

multi-step form

E. Independence score
Penalizes reliance on hints, eliminations, or guided prompts.

F. Connection score
Measures performance when the concept appears inside related problems.

8. Decay detection model
Memory Mode needs a separate system called the Decay Detection Engine.

This engine watches for patterns such as:

rising response time despite prior mastery

recent accuracy drop after a time gap

strong performance only on familiar question shapes

repeated confusion between similar concepts

more hint usage than before

previously mastered skill now failing in mixed contexts

performance collapse in dependent skills

Decay Risk Score
Create a second metric:

Decay Risk = f(time_since_last_mastery, performance_drop, speed_drop, relapse_rate, dependency_damage)

This score tells the system not just current strength, but likelihood of near-future forgetting.

A student may still be “Stable” but have high decay risk. That is when the app should intervene early.

schedule follow-up checks

Dynamic support rules
If the learner fails two questions in a row:

reduce difficulty,

add a cue,

show a smaller pattern,

or temporarily step down to a prerequisite.

If the learner succeeds three times in a row:

remove cues,

increase variation,

introduce mixed context,

then validate under time.

That is not how memory behaves. A useful way to model it is the distinction between storage strength and retrieval strength: something may be strongly learned but temporarily hard to access, while something learned very recently may feel easy right now only because it is still highly accessible, not because it is deeply stored. Retrieval strength drops with time much faster when storage strength is still low. 

The second big insight is this: you cannot prove memory from one correct answer. One correct answer may come from recognition, luck, short-term freshness, or cueing. What you really want is repeated evidence that the student can retrieve, use, and keep retrieving the idea after time has passed. Research on retrieval practice and spacing shows that long-term retention improves when students repeatedly retrieve information over time, not just restudy it, and that the best gap between reviews depends on how far into the future you want the knowledge retained. 

So the product should stop asking:

“Did the student get it right?”

and start asking:

“Did the student produce enough evidence that this knowledge is now durable?”

What it means for something to be “truly in memory”
A concept should only count as truly in memory when the student can do most of these reliably:

Recall it without heavy cues

Recall it again after a delay

Recall it when the question changes form

Use it inside a connected problem

Do it with reasonable speed

Survive interference from similar concepts

That is the proof.

Not one question.
Not one session.
A pattern.

The key distinction your app needs
Your app should track two separate things for every subtopic or micro-skill:

1. Memory Access
Can the student pull it out right now?

This is current retrievability.

2. Memory Entrenchment
Has this concept been reinforced enough over time and across conditions that it is likely to remain?

This is durable storage.

That distinction matters because a student may look good today and still forget tomorrow. Bjork’s framework captures this well: recent success can reflect high retrieval strength without high storage strength, while deeply learned material may sometimes become temporarily hard to retrieve but come back quickly on relearning.

Weak evidence of memory

These may show short-term accessibility, not durable memory.

Stronger evidence of memory

explain-why questions

reverse questions

same skill in a different context

mixed-topic questions where the student must identify the right concept first

delayed recall questions after time has passed

Research on retrieval practice shows that successful retrieval does more than reveal memory — it strengthens later recall, and repeated testing tends to help delayed recall more than repeated restudy.

A skill should not be marked “in memory” after one success.

This checks whether the memory is usable.

This checks whether the memory is stable, not accidental.

3. Canonical memory states
Use these states:

SEEN

ENCODED

ACCESSIBLE

FRAGILE

ANCHORING

CONFIRMED

LOCKED_IN

AT_RISK

FADING

REBUILDING

RECOVERED

COLLAPSED

Meaning:

SEEN
Student encountered it, but no retrieval proof yet.

ENCODED
Initial lesson or practice happened, but recall evidence is still shallow.

ACCESSIBLE
Student can currently retrieve it in at least one low-friction context.

FRAGILE
Some evidence exists, but not enough to trust durability.

ANCHORING
Multiple evidence dimensions are filling in, but time-separated proof is incomplete.

CONFIRMED
Enough evidence exists across required dimensions and time bands.

LOCKED_IN
Confirmed plus repeated long-interval stability.

AT_RISK
Still mostly retrievable, but decay risk is rising.

FADING
Clear retrieval weakening or interference breakdown.

REBUILDING
System is actively repairing the skill.

RECOVERED
Skill came back after fading, but needs monitoring.

COLLAPSED
Recall evidence has broken badly enough that the system should not trust access.

4. State transition model
4.1 Upward transitions
SEEN -> ENCODED

learner completes instruction or first guided exposure

ENCODED -> ACCESSIBLE

first successful independent or lightly cued retrieval

ACCESSIBLE -> FRAGILE

one or more successful retrievals, but no time-separated or transfer proof yet

FRAGILE -> ANCHORING

at least two proof dimensions satisfied

at least one distinct variant success

error rate dropping

ANCHORING -> CONFIRMED

proof template satisfied

minimum delay success satisfied

interference not unresolved

current access above threshold

CONFIRMED -> LOCKED_IN

repeated success across longer interval bands

low relapse count

stable speed and transfer

4.2 Risk transitions
CONFIRMED -> AT_RISK

due for recheck and delay exceeded

response-time drift up

neighboring dependent skills slipping

recent interference confusion rising

AT_RISK -> FADING

failed delayed recheck

recent accuracy collapse

repeated need for cues

mixed-context failures increase

FADING -> REBUILDING

rescue session launched

REBUILDING -> RECOVERED

rescued skill regains required local evidence

one short-delay success achieved

RECOVERED -> CONFIRMED

proof template re-satisfied with at least one time-separated success after recovery

4.3 Failure transitions
ACCESSIBLE -> COLLAPSED

repeated failures even with light cues

classifier says concept is not merely fading but functionally unavailable

FRAGILE -> COLLAPSED

high error rate across canonical, variant, and guided forms

AT_RISK -> COLLAPSED

complete breakdown on recheck and prerequisite chain damage

REBUILDING -> COLLAPSED

rescue unsuccessful and prerequisite rebuild required

5. Guard conditions for promotion
A skill should never be promoted by score alone.
Promotion requires both:

numeric threshold

proof-template coverage

That means CONFIRMED should require:

sufficient retrieval access

sufficient durability confidence

required proof dimensions filled

at least one time-separated success

at least one non-identical success

no unresolved interference block

This is important because repeated testing and spaced retrieval are strong signals for long-term retention, but the system still needs subject-specific evidence that the knowledge is usable and transferable. 

6. Main numeric scores
6.1 RetrievalAccessScore
Range: 0–100

Measures:

recent correctness

speed against expected time

cue independence

confidence calibration

Suggested form:

RAS = 35*recent_accuracy + 20*speed_score + 20*independence_score + 15*consistency_score + 10*confidence_alignment

All subcomponents normalized to 0–1.

Purpose:
Can the learner pull this out now?

6.2 DurabilityConfidenceScore
Range: 0–100

Measures:

time-separated success

spaced survival

number of distinct successful variants

embedded-use success

relearning speed after lapse

Suggested form:

DCS = 25*time_separated_score + 20*variant_score + 15*embedded_use_score + 15*interference_resistance_score + 15*recheck_stability_score + 10*relearning_efficiency

Purpose:
How much evidence says this knowledge will last?

6.3 DecayRiskScore
Range: 0–100, where higher is worse

Measures:

overdue recheck age

response-time drift

recent error drift

relapse history

low spacing success

neighboring dependency collapse

Suggested form:

DRS = 25*overdue_factor + 20*accuracy_drop + 15*speed_drift + 15*relapse_factor + 15*dependency_damage + 10*interference_growth

Purpose:
How likely is slippage soon?

6.4 InterferenceRiskScore
Range: 0–100

Measures confusion with similar concepts.

Examples:

area vs perimeter

osmosis vs diffusion

mitosis vs meiosis

11. Transition decision logic
Pseudo-logic:

def evaluate_skill(learner_id, skill_id):
    mem = load_memory_row(learner_id, skill_id)
    proof = load_proof_vector(learner_id, skill_id)
    template = load_template(skill_id)

    ras = compute_ras(mem, proof)
    dcs = compute_dcs(mem, proof)
    drs = compute_drs(mem, proof)

    template_ok = template.is_satisfied(proof)
    has_delay = proof["delayed_recall"].status == "satisfied"
    has_variant = any_dim_satisfied(proof, ["variant_transfer", "representation_shift"])
    interference_block = mem.interference_risk_score > 70 and not proof["interference_resistance"].status == "satisfied"

    if ras < 25 and dcs < 20:
        new_state = "COLLAPSED"
    elif drs > 75 and ras < 45:
        new_state = "FADING"
    elif current_rescue_open(skill_id, learner_id):
        new_state = "REBUILDING"
    elif template_ok and has_delay and has_variant and not interference_block and ras >= 70 and dcs >= 70:
        new_state = "CONFIRMED"
    elif template_ok and ras >= 80 and dcs >= 85 and stable_long_interval_successes(learner_id, skill_id):
        new_state = "LOCKED_IN"
    elif drs > 55 and ras >= 55:
        new_state = "AT_RISK"
    elif ras >= 55 and proof_dimensions_count(proof) >= 2:
        new_state = "ANCHORING"
    elif ras >= 45:
        new_state = "FRAGILE"
    elif ras >= 30:
        new_state = "ACCESSIBLE"
    else:
        new_state = "ENCODED"

    persist_state(new_state)
12. Recheck scheduler logic
Spacing should be adaptive, because the best review gap depends on how far out you want the knowledge to be retained; there is no single universal interval that is optimal for all goals. 

Suggested scheduler:

If state is FRAGILE
same day or next day

target: delayed recall

If state is ANCHORING
2–4 days

target: variant transfer or embedded use

If state is CONFIRMED
7–14 days

target: delayed recall in changed form

If state is LOCKED_IN
21–45 days

target: fast transfer check or exam-context mixed problem

If state is AT_RISK
immediate rescue or 24h recheck

If state is RECOVERED
short delay again

then medium delay

then back to normal schedule

The scheduler should also compress intervals as the exam date approaches.

13. Evidence weighting rules
Not all correct answers should count equally.

Example weight map:

identical MCQ success: 0.8

familiar short-answer success: 1.5

free recall success: 2.5

delayed recall success: 3.0

variant transfer success: 2.5

embedded-use success: 2.5

interference discrimination success: 2.5

explanation/reasoning success: 2.0

representation-shift success: 2.5

long-gap recheck success: 3.5

Lower cue support should earn more credit than heavy cue support, because reduced cueing is a stronger sign that retrieval is learner-generated rather than prompt-supported. That also aligns with findings that impoverished cue support can strengthen later retention. 

14. Recovery planner logic
When state becomes FADING or COLLAPSED, classify the breakdown.

Possible breakdown types:

PURE_DECAY

INTERFERENCE_CONFUSION

PREREQUISITE_BREAK

SURFACE_DEPENDENCE

SPEED_ONLY_WEAKNESS

UNDERSTANDING_NOT_MEMORY

Then choose the repair path.

PURE_DECAY
Use:

independent recall

short delay repeat

variant transfer

follow-up recheck

INTERFERENCE_CONFUSION
Use:

contrast drills

side-by-side discrimination

interference-specific questions

then delayed discrimination recheck

PREREQUISITE_BREAK
Repair root skill first.

SURFACE_DEPENDENCE
Use:

changed wording

changed representation

mixed context

reverse questions

SPEED_ONLY_WEAKNESS
Use:

light fluency drills

preserve concept difficulty

avoid over-remediation

UNDERSTANDING_NOT_MEMORY
Do not stay in Memory Mode.
Hand off to concept-teaching or weakness-fix mode.

17. Anti-gaming rules
The system must resist fake memory signals.

Rules:

identical question repeats give sharply diminishing weight

back-to-back same-family items capped

high cue support cannot by itself satisfy independent recall

immediate post-lesson success cannot confirm durability

only time-separated evidence can satisfy delayed proof

only distinct surface forms can satisfy transfer proof

only mixed/interference items can satisfy those respective dimensions

This matters because students often mistake ease and familiarity for lasting learning, and repeated restudy can inflate confidence more than delayed retention.
