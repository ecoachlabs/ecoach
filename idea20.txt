Academic Coach — Problems & Proposed Solutions

  PROBLEM 1: The Coach Abandons the User

  What happens: After completing any action (diagnostic, mission, onboarding), the student lands on a generic page with no clear next step. They don't know what to do.

  Root cause: The Coach Hub is a static dashboard, not a guided experience. It shows data but doesn't LEAD.

  Solution: The Coach Hub should ALWAYS show ONE clear next action at the center of the screen. No browsing, no guessing. The coach decides what's next and presents it. Every
  screen answers: "What should I do RIGHT NOW?"

  ---
  PROBLEM 2: Diagnostic Loop

  What happens: After completing a diagnostic, the student sees the same diagnostic again. Endless loop.

  Root cause: The orchestrator can't generate real missions because no topic profiles exist in the database. Placeholder questions don't produce real evidence. So the system keeps
   falling back to diagnostics.

  Fix applied: After the first mission completes, the orchestrator now seeds topic profiles from Journey content pack data (j_concept_nodes). This gives the mission planner real
  topics to work with.

  Fix applied: Phase logic simplified — once any episode exists, never show diagnostic screens again.

  Remaining risk: The seeding depends on Journey content packs being installed. If none are installed, seeding returns zero topics and the orchestrator is stuck.

  ---
  PROBLEM 3: Placeholder Questions

  What happens: All questions say "This question will be loaded from the question intelligence system." No real content.

  Root cause: The generateStepsFromBlueprint() function in MissionFlow creates placeholder StepQuestion objects because it doesn't connect to the real question database (Journey's
   j_concept_nodes, question_instances, question_families).

  Solution: MissionFlow's step generation needs to query the actual question database — load real questions from question_instances filtered by the mission's topic/subject. The
  question intelligence layer already exists (migration 031), it just needs to be wired into the coach's step generation.

  ---
  PROBLEM 4: No Study Plan

  What happens: The coach has no structured plan. It generates one-off missions reactively instead of following a deliberate day-by-day program.

  Root cause: No plan engine exists. The orchestrator picks the highest-priority topic at the moment but has no concept of a multi-day structured plan.

  Solution: Build a Plan Engine that:
  - Takes exam date + subjects + diagnostic results
  - Generates a day-by-day structured plan with mixed activities (learn, practice, test, review, speed drill, repair)
  - Specifies daily time budgets
  - Sets milestones and targets
  - Adapts after each session based on performance
  - Shows a visual timeline on the Coach Hub overview

  ---
  PROBLEM 5: Self-Reported Assessment

  What happens: RecallBurstStep asks "Did you get it right?" and the student can lie.

  Fix applied: All recall_burst steps now render as real MCQ questions. The RecallBurstStep component is no longer used.

  ---
  PROBLEM 6: No Subject Selection Before Diagnostic

  What happens: The diagnostic starts without knowing what subjects the student is studying.

  Fix applied: Added Step 2 (Subject Selection) to the onboarding flow. Also added subject selection to the first_diagnostic screen for users who skip onboarding.

  ---
  PROBLEM 7: Content Not Centered

  What happens: Screens have content crammed in the top-left with 60% of the screen empty.

  Fix applied: MissionFlow work area uses flex items-center justify-center. All coach pages have mx-auto. Onboarding steps are full-viewport centered.

  ---
  PROBLEM 8: Mode Transition Glitches

  What happens: Toggling Coach Mode causes card flickering, position shifts, and visual glitches.

  Root cause: The ripple transition overlay caused React re-renders during navigation. Store state changes (coachModeActive, transitioning) triggered layout recalculations visible
   through translucent overlays.

  Fix applied: Transition component disabled (no-op). Coach toggle now does instant navigation. The sidebar hides via coachModeActive state directly.

  Future solution: Revisit the transition effect in a dedicated session with a simpler approach — maybe just a CSS cross-fade on the main content area using the View Transition
  API, not a custom overlay.

  ---
  PROBLEM 9: Dead Code in CoachHub.tsx

  What happens: Babel parse error at line 711 — {/* Resume */} inside a multi-line /* */ comment breaks the parser because */ in JSX comments closes the outer comment.

  Root cause: I tried to comment out orphaned code with /* */ but JSX comments containing */ broke it. Then I tried if(false) { template literal } which Babel also can't parse.

  Fix needed: Delete the dead code block entirely (lines 684-856). It's wrapped in a broken comment/template literal. Just remove it and the CoachOnboarding function that follows
  is clean.

  ---
  PROBLEM 10: No Time Tracking

  What happens: There's no timer. The coach doesn't track how long the student spends studying or whether they've met their daily goal.

  Solution: Add a timer to the Coach Hub header that:
  - Shows today's remaining study budget (e.g. "35 min remaining")
  - Counts down when Coach Mode is active
  - Pauses when Coach Mode is toggled off
  - Persists elapsed time to the database
  - Uses c_engagement_events to track session starts/stops

  ---
  PROBLEM 11: No Visual Roadmap

  What happens: The student can't see where they are in their journey toward the exam. No timeline, no progress visualization.

  Solution: Build a visual timeline on the Coach Hub Overview showing:
  - Day 1 → Exam Day progress bar with "you are here" marker
  - Current preparation phase highlighted
  - Today's planned activities as horizontal cards (done ✓, current ▶, upcoming ○)
  - Weekly progress indicator
  - Readiness trajectory (current vs target)

  ---
  PROBLEM 12: No Adaptive Blocking

  What happens: The coach moves on to new topics even when the student is failing the current one.

  Solution: Add blocking rules to the mission planner:
  - If accuracy < 40% after 2 sessions on a topic → BLOCK forward progress
  - Deploy reteach + worked example + guided practice
  - Only unlock when accuracy reaches 60%
  - If misconception recurs 3+ times → force a targeted repair session

Decision order
Use strict order. Do not allow screens to override it.

function resolveCoachState(studentId: string): CoachStateResolution {
  if (!hasCompletedOnboarding(studentId)) {
    return { state: "onboarding_required", reasons: ["onboarding_incomplete"], data: {} };
  }

  if (!hasSelectedSubjects(studentId)) {
    return { state: "subject_selection_required", reasons: ["no_subjects"], data: {} };
  }

  const readiness = getContentReadiness(studentId);
  if (readiness.status !== "ready") {
    return {
      state: readiness.status === "no_packs_installed" || readiness.status === "no_topics_available"
        ? "stalled_no_content"
        : "content_readiness_required",
      reasons: [readiness.status],
      data: { readiness }
    };
  }

  if (hasPendingMissionReview(studentId)) {
    return { state: "mission_review_required", reasons: ["pending_review"], data: {} };
  }

  if (hasActiveMission(studentId)) {
    return { state: "mission_in_progress", reasons: ["active_mission"], data: {} };
  }

  if (requiresRepair(studentId)) {
    return { state: "repair_required", reasons: ["repair_needed"], data: {} };
  }

  if (hasBlockedTopic(studentId)) {
    return { state: "blocked_on_topic", reasons: ["topic_blocked"], data: {} };
  }

  if (!hasSufficientBaselineEvidence(studentId)) {
    return { state: "diagnostic_required", reasons: ["insufficient_baseline"], data: {} };
  }

  if (!hasActivePlan(studentId)) {
    return { state: "plan_generation_required", reasons: ["no_active_plan"], data: {} };
  }

  if (planNeedsAdjustment(studentId)) {
    return { state: "plan_adjustment_required", reasons: ["plan_stale_or_changed"], data: {} };
  }

  if (isExamMode(studentId)) {
    return { state: "exam_mode", reasons: ["exam_window"], data: {} };
  }

  if (isReviewDay(studentId)) {
    return { state: "review_day", reasons: ["review_schedule"], data: {} };
  }

  return { state: "ready_for_today_mission", reasons: ["plan_ready"], data: {} };
}

function generateCoachPlan(input: PlanInput): CoachPlan {
  const daysRemaining = diffInDays(input.currentDate, input.examDate);
  const subjectWeights = allocateSubjectTime(input.subjects, input.evidence);
  const topicBuckets = classifyTopics(input.topicProfiles);

  const phaseMap = buildPhaseMap(daysRemaining);
  const planDays = [];

  for (const day of eachDayUntilExam(input.currentDate, input.examDate)) {
    const phase = phaseMap[day];
    const budget = input.dailyBudgetMinutes;

    const activities = buildDailyActivities({
      phase,
      budget,
      subjectWeights,
      topicBuckets,
      evidence: input.evidence
    });

    planDays.push(createPlanDay(day, phase, budget, activities));
  }

  return persistPlan(planDays);
}

4. The brain’s major engines
The brain should be decomposed into 10 engines.

4.1 Perception Engine
This engine collects raw signals and normalizes them.

Inputs include:

onboarding completion

subjects selected

exam target

exam date

diagnostics completed

question answer events

timer events

mission start/stop events

review events

parent feedback

content pack installation state

question coverage state

manual overrides

Outputs include:

normalized event stream

latest session facts

usage facts

content facts

topic exposure facts

missing prerequisite facts

This engine does not decide anything.
It only converts raw signals into usable observations.

4.2 Student Model Engine
This engine constructs the learner profile.

It should maintain academic traits such as:

pace of learning

stability under time pressure

concept recognition strength

application strength

transfer ability

misconception tendencies

persistence after failure

need for worked examples

need for repeated reinforcement

tendency to rush

tendency to hesitate

A student model output might say:

strong in untimed concept recall, weak in timed application

repeated operator confusion in algebra

accuracy acceptable but latency too high for exam conditions

benefits from guided examples before independent practice

This engine should not produce vague narratives only.
It should produce structured signals used elsewhere.

Example:

type StudentLearningProfile = {
  paceProfile: "slow" | "steady" | "fast";
  pressureResponse: "stable" | "fragile" | "collapses_under_timer";
  conceptVsApplication:
    | "concept_stronger"
    | "balanced"
    | "application_stronger";
  hintDependency: "low" | "medium" | "high";
  misconceptionClusters: string[];
  recoveryAfterFailure: "good" | "moderate" | "poor";
};
4.3 Curriculum Intelligence Engine
This engine is the academic map.

It should know:

subject hierarchy

topic hierarchy

subtopic hierarchy

prerequisites

dependencies

concept-to-formula relationships

common confusion pairs

misconception families

question family mappings

what future topics depend on which earlier ones

This is not just storage. It is reasoning-ready curriculum structure.

Example:

If the student is weak in fractions, this engine should know that this can affect:

percentages

ratio

algebraic fractions

probability representations

certain forms of mensuration and data interpretation

That allows the brain to prioritize earlier repair before later damage spreads.

4.4 Evidence Engine
This is where performance becomes truth.

It should update topic state from actual behavior, not just scores.

It should compute:

correct count

accuracy

average latency

hint usage rate

retry rate

first-pass success

transfer success

misconception recurrence

exposure count

recency decay

confidence of estimate

4.5 Lifecycle State Engine
This engine determines the canonical coach state.

It resolves whether the student is currently in:

diagnostic

repair

plan adjustment state

exam mode

This is deterministic and strict.
It must be fully unit-tested.

4.6 Plan Engine
This is the strategic brain.

It must generate:

plan phases

dated plan days

daily activity blocks

subject distribution

topic sequencing

checkpoint placement

review and repair reservations

catch-up logic

pre-exam consolidation behavior

The plan engine must reason over:

days remaining

study budget

topic weakness

fragility

coverage availability

current phase

backlog

exam proximity

This engine should not generate a vague plan.
It should generate actual executable plan entities.

4.7 Mission Planner
This is the tactical engine.

It must answer:

what session should the student do now

why that session exists

what topic it covers

what activity type it should be

what question/data resources are required

what counts as success or failure

how long it should last

The mission planner should never invent content.
It must request real content through the content/question adapter.

4.8 Intervention Engine
This is the live coaching instinct.

It should respond to patterns like:

repeated misconception

low accuracy but good effort

high accuracy but too slow

correct method choice but arithmetic slips

good recall, poor transfer

speed collapse after timer activation

frustration after repeated wrong answers

It should choose interventions like:

worked example replay

misconception spotlight

comparison drill

guided breakdown

slowed mode

repair burst

restore confidence prompt

defer speed training until stability returns

This engine can use bounded reasoning because intervention choice benefits from nuance.

4.9 Synchronization Engine
This keeps everything aligned.

When something changes, this engine determines all downstream updates.

Example after mission completion:

persist mission result

persist question-level results

update evidence state

update topic profiles

update blocker status

update repair obligations

update plan progress

update time budget

resolve new lifecycle state

resolve new next action

regenerate hub payload

regenerate roadmap payload

generate debrief payload

trigger parent update if policy says so

This engine is what prevents fragmentation.

4.10 Explainability Engine
The brain must be able to explain itself in clean structured language.

It should answer:

why is this the next action

why is this topic blocked

why was a repair inserted

why did today’s strategy change

why was the student classified as fragile here

why is the coach delaying timed mode

Output format:

  whyThisState: string[];
  whyThisAction: string[];
  whyThisStrategy: string[];
  risks: string[];

5. The central evaluation loop
The brain should run a controlled evaluation loop.

This is the heartbeat of the coaching system.

Step 1 — Observe
Collect latest events and current storage state.

Step 2 — Build world state
Assemble CoachWorldState.

Step 3 — Resolve kernel truth
Determine readiness, lifecycle state, plan validity, blockers, time truth.

Step 4 — Run intelligence engines
Interpret evidence, learner pattern, strategy options, intervention need.

Step 5 — Decide next action
Resolve what should happen now.

Step 6 — Generate outputs
Mission payload, hub payload, roadmap payload, timer payload, debrief payload.

Step 7 — Persist changes
Store updated snapshots, plan adjustments, evidence, blockers, etc.

Step 8 — Dispatch sync actions
Update every downstream consumer.

Step 9 — Explain
Produce structured reasons.

A top-level function:

async function evaluateCoachBrain(studentId: string, trigger: BrainTrigger): Promise<CoachBrainOutput> {
  const observations = await perceptionService.collect(studentId, trigger);
  const worldState = await worldStateBuilder.build(studentId, observations);

  const lifecycle = lifecycleStateEngine.resolve(worldState);
  const learnerProfile = studentModelEngine.derive(worldState);
  const evidenceSummary = evidenceEngine.derive(worldState);

  const planResult = await planEngine.evaluate(worldState, learnerProfile, evidenceSummary);
  const missionResult = await missionPlanner.evaluate(worldState, planResult);
  const interventionResult = interventionEngine.evaluate(worldState, missionResult, learnerProfile);

  const nextAction = nextActionResolver.resolve(worldState, lifecycle, planResult, missionResult, interventionResult);
  const syncActions = synchronizationEngine.build(worldState, lifecycle, nextAction, planResult, missionResult, interventionResult);

  const explanation = explainabilityEngine.explain(worldState, lifecycle, nextAction, planResult, missionResult, interventionResult);

  return {
    worldState,
    currentState: lifecycle.state,
    nextAction,
    todayStrategy: planResult.todayStrategy,
    mission: missionResult.mission,
    intervention: interventionResult,
    roadmap: planResult.roadmap,
    syncActions,
    explanation
  };
}

12. Topic state scoring
The topic profile should not be a vague label only.
It needs real numbers.

A workable first version:

Mastery estimate
Combination of:

recent accuracy

stability across exposures

transfer success

recency weighting

Fragility score
High when:

timed performance drops sharply vs untimed

spaced recall collapses

accuracy varies wildly across sessions

Speed score
High when:

correctness remains strong under time

latency is within acceptable band

panic errors are low

Repair priority
High when:

topic is prerequisite-heavy

topic is blocking downstream concepts

misconception recurrence is high

exam relevance is high

recent failures are persistent

A simple conceptual formula:

masteryEstimate =
  0.35 * recentAccuracy +
  0.20 * spacedStability +
  0.20 * transferSuccess +
  0.15 * firstPassQuality +
  0.10 * recencyWeightedConfidence;

fragilityScore =
  0.45 * pressureDrop +
  0.25 * varianceAcrossSessions +
  0.20 * retentionDrop +
  0.10 * hesitationRate;

speedScore =
  0.50 * timedAccuracy +
  0.30 * latencyEfficiency +
  0.20 * lowPanicErrorRate;
These can evolve, but the structure matters.

13. Blocking and repair logic
The brain must stop false progress.

Block rule example
A topic becomes blocked if all are true:

at least 2 completed sessions

attempts threshold met

accuracy below 40%

misconception recurrence above threshold

trend non-improving

Unblock rule example
A topic becomes unblocked if:

repair completed

guided accuracy >= 60%

misconception absent in checkpoint tail

no severe collapse under mild timing

Repair trigger example
Trigger targeted repair if any are true:

same misconception repeated 3 times

untimed okay, timed disastrous

strong recognition, weak application

repeated family-variant failure

high-confidence wrong answers recurring

The blocker engine should be deterministic.
The intervention style can be adaptive.

9.2 Policy categories

do not schedule mixed challenge if foundational confusion is severe

Emotional / motivation policies
do not show high-pressure experiences after repeated immediate failure

do not stack 3 hard capabilities back-to-back

Time policies
do not compose sessions longer than available minutes

avoid long warm-ups when behind schedule

Product policies
do not repeat the same feature too often without reason

do not surface roadmap too frequently as noise

do not overuse parent escalation

prefer embedded experiences over full navigation when friction is high

Capability policies
capability A may require capability B first

capability C must not be used after capability D in the same session

capability E has max uses per week

1. Purpose
The Coach Brain can only orchestrate what it knows exists.

If the product keeps growing and the registry is updated manually, the registry will drift, and the brain will become blind to new powers in the codebase.

So you need a subsystem that continuously answers:

what features currently exist in the codebase

2. Design principle
Do not rely on one method only.

A strong discovery system should use four complementary discovery modes:

A. Static code discovery
Scan files, routes, components, imports, services, hooks, schemas, registries, and state machines.

B. Metadata-assisted discovery
Use explicit capability metadata in code where available.

C. Runtime discovery
Observe actual rendered surfaces, event flows, feature activations, and instrumentation.

D. Change detection and governance
Compare scans over time and force new capability review when code changes introduce new feature-level behavior.

That combined approach is what makes this robust.

27. Suggested scan-to-registry flow in pseudocode
async function runCapabilityDiscoveryScan(scanMode: "incremental" | "full") {
  const artifacts = await codebaseScanner.scan(scanMode);
  const featureCandidates = featureCandidateExtractor.extract(artifacts);
  const enrichedCandidates = capabilitySignalEnricher.enrich(featureCandidates);

  const mapped = capabilityMapper.map(enrichedCandidates);
  const scored = confidenceScorer.score(mapped);

  const diff = diffEngine.compareWithPrevious(scored);
  const drift = driftDetector.detect(scored);

  const { autoApproved, needsReview } = splitByConfidenceAndPolicy(scored);

  await inventoryRepository.storeScan({
    artifacts,
    candidates: scored,
    diff,
    drift
  });

  await registrySyncEngine.sync(autoApproved);
  await reviewQueueService.enqueue(needsReview);

  return {
    autoApproved,
    needsReview,
    diff,
    drift
  };
}

29. Non-negotiable rules
The capability inventory must not be purely manual.

The scanner must not overwrite curated academic meaning blindly.

New coach-facing features must be discoverable or annotated.

Diff and drift detection must run continuously.

High-confidence structural fields can auto-sync.

Low-confidence academic purpose fields require review.

The brain consumes only resolved capabilities, not raw scan artifacts.

Runtime evidence should influence confidence and relationships over time.

30. Final architecture statement
What you need is:

a living capability inventory system

static discovery + metadata + runtime reconciliation + CI governance

So the Coach Brain always knows what the codebase can actually do, keeps learning as new features are added, and can orchestrate the full product intelligently without registry drift.
