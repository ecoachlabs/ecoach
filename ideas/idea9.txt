Good. So this is not answer format.
This is cognitive question type — the kind of mental work the question forces the student to do.

A strong educational system should have a question-type taxonomy like this:

1. Pure Recall Questions
Pull a fact, formula, definition, date, rule, or term straight from memory.

2. Recognition Questions
The student does not generate the answer from scratch but must identify the correct idea, concept, or statement when they see it.

3. Memory Reconstruction Questions
The student must rebuild a forgotten idea from fragments, clues, related concepts, or partial prompts.

4. Retrieval Under Pressure Questions
Same as recall, but under time pressure, repetition, or rapid-fire conditions to test mental sharpness.

5. Concept Understanding Questions
Tests whether the student truly understands what something means, not just whether they memorized it.

6. Explanation Questions
The student must explain a concept, process, or answer in their own words.

7. Reasoning Questions
The student must think through why something is true, false, correct, or possible.

8. Logical Deduction Questions
The answer must be reached by step-by-step logic from given facts or conditions.

9. Inference Questions
The student must derive what is implied, even if it is not directly stated.

10. Application Questions
Use a known principle, formula, method, or rule in a new but familiar situation.

11. Transfer Questions
Use what was learned in one context to solve a different-looking problem.

12. Multi-Step Problem Solving Questions
The student must go through several reasoning or calculation stages before reaching the answer.

13. Strategy Selection Questions
The student must decide how to solve the problem before solving it.

14. First-Step Questions
Instead of solving the whole problem, the student identifies the best first move.

15. Next-Step Questions
The student is shown a partially solved problem and must determine what should happen next.

16. Error Detection Questions
The student spots the mistake in a worked solution, statement, argument, or process.

17. Correction Questions
The student not only spots the error but fixes it properly.

18. Misconception Exposure Questions
Designed to reveal common misunderstandings and trap shallow thinking.

19. Compare-and-Contrast Questions
The student must identify similarities, differences, boundaries, or distinctions between ideas.

20. Classification Questions
The student places an item, example, event, or idea into the right category, group, or concept family.

21. Matching Principle-to-Case Questions
The student must connect the right rule, theorem, concept, law, or method to the right scenario.

22. Cause-and-Effect Questions
Tests whether the student understands what leads to what, and why.

23. Sequence / Order Questions
The student arranges ideas, events, steps, or processes in the correct order.

24. Prediction Questions
Given a situation, the student predicts the likely outcome, next event, or result.

25. Estimation Questions
The student gives a sensible approximate answer using reasoning rather than exact computation.

26. Pattern Recognition Questions
The student identifies trends, structures, relationships, or hidden regularities.

27. Rule Discovery Questions
The student is not given the rule directly; they must infer the rule from examples.

28. Abstraction Questions
The student must move from a concrete example to a general principle or deeper idea.

29. Example Generation Questions
The student creates their own valid example of a concept, rule, structure, or phenomenon.

30. Non-Example Questions
The student identifies or creates something that looks close, but does not satisfy the concept.

31. Justification Questions
The student must defend an answer with reasoning, evidence, or steps.

32. Evidence Selection Questions
The student chooses which fact, data point, theorem, or clue best supports a conclusion.

33. Claim Evaluation Questions
The student decides whether a statement is valid, weak, exaggerated, unsupported, or false.

34. Counterexample Questions
The student disproves a claim by producing a valid exception.

35. Decision-Making Questions
The student chooses the best option among several and explains why it is best.

36. Prioritization Questions
The student decides which factor, step, variable, or issue matters most.

37. Synthesis Questions
The student combines multiple ideas or topics to form one answer.

38. Connection-Making Questions
The student links ideas across topics, chapters, or subjects.

39. Interpretation Questions
The student must interpret a diagram, graph, passage, experiment, data set, or scenario.

40. Representation Conversion Questions
The student converts one form of knowledge into another:
word to equation, equation to graph, graph to explanation, diagram to rule.

41. Visualization Questions
The student must mentally picture shape, motion, arrangement, structure, or process.

42. Mental Manipulation Questions
The student rotates, rearranges, transforms, simplifies, or tracks things in the mind.

43. Attention Control Questions
These test whether the student can stay alert and avoid traps, distractions, or overlooked details.

44. Precision Questions
The student must answer carefully with exactness, correct wording, correct unit, or strict logical boundary.

45. Speed Fluency Questions
Measures how quickly the student can retrieve or process well-known material accurately.

46. Deep Thinking Questions
These are deliberately thought-provoking and require reflection, layered reasoning, or insight.

47. Open-Ended Reasoning Questions
There may be more than one acceptable path, but the reasoning must be strong.

48. Real-World Scenario Questions
The concept is embedded inside a realistic situation so the student must think beyond textbook wording.

49. Judgment Questions
The student must evaluate quality, validity, efficiency, fairness, plausibility, or correctness.

50. Reflection / Metacognitive Questions
The student thinks about their own thinking:
Why did I choose this?
Where did I get stuck?
What confused me?

51. Confidence Calibration Questions
The student answers and also indicates how sure they are, helping detect guessing vs mastery.

52. Mastery Check Questions
Questions designed not just to see if they can answer once, but whether the knowledge is stable and repeatable.

53. Retention Check Questions
Used after time has passed to see whether knowledge actually stayed in memory.

54. Recovery Questions
Given when memory is fading, to revive old knowledge and rebuild weak recall pathways.

55. Adaptive Difficulty Questions
The question adjusts based on the learner’s current strength, weakness, speed, and memory profile.

56. Challenge / Stretch Questions
Within the syllabus, but harder, deeper, trickier, or more layered for elite learners.

57. Rescue Questions
Simplified but intelligent questions meant to rebuild confidence and re-enter understanding when a learner is struggling badly.

58. Diagnostic Questions
Questions whose main purpose is to discover what exactly the student does or does not understand.

59. Threshold Questions
These determine whether the student is ready to move to the next level, topic, or skill.

60. Capstone Questions
Bigger, richer questions that combine many subskills and show full command.

Yes — this is the right next layer.

A question factory cannot generate strong cognitive question types by “just writing questions.”
It has to operate like a structured transformation engine.

The core idea is this:

Every question is generated from a knowledge object plus a cognitive demand plus a delivery condition.

So if the system knows:

what the learner should know

what kind of thinking you want to force

how hard, fast, tricky, or diagnostic it should be

then it can generate many question types reliably.

1. The hidden truth behind question generation
Your engine should not begin from “write a question.”

It should begin from:

What is the target knowledge or skill?

What mental action do I want to force?

What proof will convince me the learner truly has it?

What variant of this proof fits this learner right now?

That means your question factory needs 5 foundational layers:

A. Knowledge Layer
Stores what is to be learned.

B. Cognitive Demand Layer
Stores what kind of thinking to force.

C. Transformation Layer
Turns knowledge into different question types.

D. Surface Layer
Wraps the question in wording, format, media, timing, and context.

E. Validation Layer
Checks that the generated question is valid, fair, clear, and diagnostically useful.

So the engine is not storing just “the answer.”

It is storing the structure of understanding.

3. The real unit of generation: evaluative intent
Before generating any question, the engine must decide:

What exactly am I trying to prove about this learner?

That is the evaluative intent.

Examples:

Can the student recall the formula?

Can the student recognize it when seen?

Can the student rebuild it from clues?

Can the student apply it to a familiar case?

Can the student transfer it to a disguised case?

Can the student detect an error in its use?

Can the student justify why it applies?

Can the student hold it under time pressure?

Can the student retain it after delay?

This means one concept can generate many question families.

6. The transformation logic behind question types
This is the most important part.

A cognitive question type is generated by applying a specific transformation pattern to a concept.

Below is the core logic.

A. Recall questions
Purpose
Test direct retrieval from memory.

Generation logic
Take an atomic fact, formula, term, rule, label, or step and hide it.

Operator
ASK_DIRECT(K.fact)

Example transformation
Knowledge:

“Chlorophyll helps absorb light energy.”

Generated recall:

“What pigment absorbs light energy in plants?”

Difficulty controls
easier: exact classroom wording

harder: paraphrase wording

harder still: no context cue

B. Recognition questions
Purpose
Test whether the learner can identify the correct idea when shown possibilities.

Generation logic
Place the target among near neighbors.

Operator
SELECT_TRUE(target, distractors_close_to_target)

Difficulty controls
weak distractors = easy

semantically close distractors = harder

all options plausible = hardest

C. Memory reconstruction questions
Purpose
Test whether the learner can rebuild memory from fragments.

Generation logic
Do not ask directly.
Provide connected cues, partial structures, linked concepts, or incomplete patterns.

Operator
REBUILD(target) FROM clues/prerequisites/associations

Example
Instead of:

“State the formula for area of a triangle.”

Ask:

“A triangle is half of what familiar shape’s area model?”

then derive toward the formula

Useful when
The learner has partial memory but weak retrieval strength.

D. Reasoning questions
Purpose
Force the learner to connect cause, logic, and principle.

Generation logic
Present a condition or claim and require the underlying why.

Operator
GIVEN premise -> infer principle / explanation

Example
“Why does increasing surface area speed up dissolving?”

This is not recall of a sentence.
It forces a causal chain.

Harder versions
remove obvious cues

use novel case

ask which explanation is strongest

ask which assumption must be true

E. Thought-provoking / deep thinking questions
Purpose
Force layered reflection, not surface memory.

Generation logic
Create tension, anomaly, contrast, unexpected result, or conflicting evidence.

Operator
CREATE cognitive conflict -> require resolution

Example
“A plant has water and carbon dioxide but is kept in darkness. It stays alive for some time but does not grow well. Explain why.”

This works because it is not a plain textbook sentence.
It creates a mental puzzle.

Recipe
Deep questions often come from:

exception

tension

hidden assumption

boundary case

conflicting evidence

comparison of close ideas

F. Application questions
Purpose
See if the learner can use knowledge in a case.

Generation logic
Take concept and wrap it inside a scenario.

Operator
MAP principle -> scenario

Example
Concept: density
Question: “Why does oil float on water in a spill?”

Harder versions
reduce explicit mention of concept

use unfamiliar context

mix with irrelevant details

G. Transfer questions
Purpose
Test whether the student can use the same idea in a different-looking context.

Generation logic
Keep same underlying principle, change the surface drastically.

Operator
PRESERVE deep structure; CHANGE surface form

Example
A student learned ratio from recipes.
Transfer question uses map scale or gear teeth.

This is one of the best mastery tests.
A learner who only memorized classroom patterns often fails here.

H. Multi-step problem-solving questions
Purpose
Test chained reasoning.

Generation logic
Build a path of dependent sub-decisions.

Operator
Solve = Step1 -> Step2 -> Step3 -> final

Difficulty controls
number of steps

branch points

distractor steps

hidden intermediate values

representation changes across steps

Important
The engine should store not only the final answer, but the solution graph.

Without that, it cannot generate:

first-step questions

next-step questions

error diagnosis

partial credit

thinking maps

I. First-step questions
Purpose
Test whether the learner knows how to begin.

Generation logic
Generate a full solvable problem, but ask only for the best opening move.

Operator
STOP before solving; ASK initial strategic action

Example
“What should you do first to solve this equation?”

This is excellent for diagnosing whether students are lost at entry, not just at execution.

J. Next-step questions
Purpose
Test continuity of thinking.

Generation logic
Show partial work or partial reasoning, then freeze and ask what should follow.

Operator
PARTIAL_STATE -> ASK next valid move

Best use
Math, science processes, grammar transformations, experiment reasoning.

K. Error detection questions
Purpose
Check whether the learner can spot wrong reasoning.

Generation logic
Generate a correct worked path, then intentionally corrupt one step.

Operator
INSERT_ERROR(solution_path, misconception_type)

Example error types
arithmetic slip

wrong theorem selection

illegal algebraic move

false generalization

reversed cause and effect

definition confusion

Hidden power
This lets the engine diagnose what kind of wrongness the student understands.

L. Correction questions
Purpose
Go beyond spotting error to fixing it.

Generation logic
Same as error detection, but require repaired reasoning.

Operator
DETECT(error) + REPAIR(path)

Stronger than detection
Because some learners know “something is wrong” but cannot rebuild the correct path.

M. Misconception exposure questions
Purpose
Reveal specific false beliefs.

Generation logic
Use known misconception patterns as distractor magnets.

Operator
TRIGGER misconception M deliberately

Example
Misconception:

“Heavier things fall faster.”

Question:

set up situation where this false idea feels tempting

Requirement
Your engine must maintain a misconception library per concept.

Without this, it cannot generate strong diagnostic questions.

N. Compare-and-contrast questions
Purpose
Test conceptual boundaries.

Generation logic
Choose two close ideas and ask the learner to distinguish.

Operator
SELECT adjacent concepts -> ask boundary differences

Example
evaporation vs boiling

mass vs weight

speed vs velocity

This is important because real understanding often sits at boundaries.

O. Classification questions
Purpose
See whether the student knows where examples belong.

Generation logic
Provide cases and ask which concept, category, or principle they instantiate.

Operator
CASE -> CATEGORY

Harder version
Use hybrid or deceptive examples.

P. Cause-and-effect questions
Purpose
Test causal understanding.

Generation logic
Store causal graph and ask forward or backward.

Operators
CAUSE -> EFFECT

EFFECT -> possible CAUSE

REMOVE cause -> predict change

Example
“What happens to transpiration if humidity rises?”

Q. Prediction questions
Purpose
Test dynamic understanding.

Generation logic
Start from a situation and simulate what follows.

Operator
STATE + rule -> future outcome

Strong version
Ask not just “what” but “why.”

R. Pattern recognition questions
Purpose
See whether the learner notices regularity.

Generation logic
Generate instances that share hidden structure.

Operator
FIND invariant / relation across examples

Examples
number patterns

experimental trends

grammar patterns

sequence logic

S. Rule discovery questions
Purpose
Learner infers the rule instead of being given it.

Generation logic
Provide carefully chosen examples and non-examples.

Operator
EXAMPLES + NONEXAMPLES -> infer governing rule

This is one of the strongest engines for deeper learning.
T. Example generation questions
Purpose
Test productive understanding.

Generation logic
Ask learner to produce an original valid instance.

Operator
CONCEPT -> generate valid instance

Important
The engine must know validity constraints to score this properly.

U. Non-example questions
Purpose
Test boundary understanding.

Generation logic
Ask learner to identify or create something that fails the concept.

Operator
CONCEPT -> produce near miss

This is very powerful for conceptual clarity.

V. Justification questions
Purpose
Test whether the student can defend reasoning.

Generation logic
Require answer plus supporting logic, principle, or evidence.

Operator
ANSWER + justify with rule/evidence/step

Stronger version
Require:

claim

evidence

reasoning link

W. Interpretation questions
Purpose
Test whether learner can read meaning from data/graphs/diagrams/scenarios.

Generation logic
Store one concept in multiple representational forms and convert across them.

Operator
REPRESENTATION_A -> infer meaning

Example
graph -> verbal explanation

X. Representation conversion questions
Purpose
See if knowledge survives form change.

Generation logic
Convert same concept across forms.

Operator
text -> equation

equation -> graph

graph -> verbal

process -> diagram

This is great for real mastery.

Y. Retrieval under pressure / mental questions
Purpose
Test recall speed and mental sharpness.

Generation logic
Take retrieval items and deliver them in timed burst sequences.

Operator
RECALL + reduced response window + streak structure

Important
Pressure itself becomes part of the question condition.

Difficulty can rise by:

shorter time

more items in sequence

mixed topics

no pause between items

interruption recovery

Z. Retention and recovery questions
Purpose
Check memory stability after delay, then rebuild weak paths.

Generation logic
Reuse concept later with changed phrasing, changed form, or lower cue level.

Operators
DELAYED_RECALL

DELAYED_TRANSFER

RECOVERY_FROM_CUES

Hidden power
These questions are not only about correctness.
They help estimate memory strength.

7. The universal generation operators
Instead of coding each question type from scratch, your engine can rely on reusable transformation operators.

These are the real building blocks.

Core operators
Hide: remove the target fact/step/conclusion

Mask: partially conceal parts of a known structure

Paraphrase: change wording while preserving meaning

Reorder: shuffle sequence or logic

Perturb: change values/context but preserve deep structure

Insert Error: place misconception or invalid step

Contrast: place adjacent ideas side by side

Transfer: move principle to new context

Compress Time: force quick response

Expand Explanation: require justification

Switch Representation: graph/text/equation/diagram conversion

Reduce Cues: make retrieval harder

Add Distractors: increase discrimination demand

Fragment: turn full knowledge into clues

Chain: combine steps

Gate: ask for first step, next step, or threshold proof only

Delay: resurface after time for retention testing

Wrap in Scenario: embed in real-life context

Introduce Conflict: create anomaly or tension

Mirror Misconception: bait common wrong belief

Reverse Reasoning: go from result back to cause or rule

Once these operators exist, the engine can generate many families systematically.

10. Fresh question generation without losing control
A major issue is freshness.

You want new questions, but not random nonsense.

The right way is controlled variation.

Freshness methods
1. Parameter variation
Change numbers, names, objects, cases.

2. Surface paraphrase
Different wording, same intent.

3. Context recasting
Same deep structure, different scenario.

4. Representation shift
Same idea in table, graph, story, symbolic form.

5. Step slicing
Ask first step, middle step, full step, or repair step.

6. Distractor mutation
Swap misconception bait.

7. Cue adjustment
More or fewer hints.

8. Difficulty band shift
Same family, different strength.

9. Time condition shift
Untimed, lightly timed, pressure mode.

10. Memory distance shift
Immediate, 2-day, 1-week, 1-month resurfacing.

So freshness is not chaos.
It is structured recombination.

17. The generation algorithm
Here is a clean pseudocode version.

def generate_question(student, concept, goal):
    learner_state = get_learner_state(student, concept)
    cognitive_type = select_cognitive_type(goal, learner_state, concept)
    family = get_question_family(concept, cognitive_type)

    primitives = fetch_concept_primitives(concept)
    misconceptions = fetch_misconceptions(concept)
    representations = fetch_representations(concept)

    difficulty = compute_target_difficulty(student, concept, goal)
    variant_plan = build_variant_plan(cognitive_type, difficulty, learner_state)

    raw_item = transform(
        family=family,
        primitives=primitives,
        misconceptions=misconceptions,
        representations=representations,
        variant_plan=variant_plan
    )

    rendered_item = render_surface(
        raw_item,
        format=choose_format(cognitive_type, learner_state, goal),
        timing=choose_timing(goal, learner_state),
        language_level=student.language_level
    )

    validated_item = validate(rendered_item, concept, cognitive_type)

    return validated_item
That is the general engine.

21. What makes “thought-provoking” questions possible
You asked specifically about reasoning and thought-provoking types.

These come from 7 patterns:

Something surprising happens. Explain it.

2. Conflict pattern

Case sits at the edge of a rule. Decide whether rule still applies.

4. Hidden assumption pattern
A conclusion seems right until the missing assumption is exposed.

5. Counterexample pattern
Disprove a sweeping claim.

6. Reverse reasoning pattern
Given the outcome, infer the cause or missing condition.

7. Multiple-path pattern
Two methods seem possible. Which is better and why?

If your engine explicitly stores these patterns, it can generate real thinking questions rather than just hard textbook questions.

2. The architecture at a glance
The full engine can be thought of as 10 connected layers:

Layer 1 — Curriculum and Knowledge Layer
Defines the syllabus, topics, subtopics, skills, concepts, prerequisites, and learning objectives.

6. How difficulty should be modeled
Do not use a crude easy-medium-hard system alone.

Each generated question should get a structured difficulty profile.

Difficulty dimensions
Concept difficulty
How intrinsically hard is the concept?

Retrieval difficulty
How much must be recalled with little support?

Distractor difficulty
How close are wrong options to the truth?

Representation difficulty
How hard is the form in which the concept is presented?

Process difficulty
How many steps or logical transitions are needed?

Context difficulty
How disguised or unfamiliar is the scenario?

Explanation difficulty
How much articulation is required?

Pressure difficulty
How much time stress is present?

Trap difficulty
How strong is the misconception bait?

A sample internal model:

difficulty_profile = {
  concept: 0.55,
  retrieval: 0.80,
  distractor: 0.30,
  representation: 0.40,
  process: 0.20,
  context: 0.10,
  explanation: 0.00,
  pressure: 0.75,
  trap: 0.25
}
This lets the system create, for example:

an easy concept but hard pressure question

a medium concept with elite-level transfer disguise

a low-step question with very strong misconception bait

That is much more intelligent.

7. How freshness should work
Freshness must be controlled.

The system should never simply “rewrite randomly.”

Use structured freshness operations:

7.1 Surface paraphrase
Change wording, preserve meaning.

7.2 Context recast
Use a different scenario for same principle.

7.3 Parameter mutation
Change names, values, variables, objects.

7.4 Representation shift
Ask same idea in graph, text, table, or verbal form.

7.5 Distractor mutation
Swap misconception bait or plausible wrong option family.

7.6 Cue shift
More support or fewer clues.

7.7 Step slice shift
Ask full problem, first step, next step, or repair step.

7.8 Time condition shift
Untimed today, rapid tomorrow.

7.9 Contrast shift
Compare with a different close concept.

Each generated question should have a freshness signature so near-duplicates can be avoided.

8. How the learner model controls generation
This is where the engine becomes truly adaptive.

For each concept, the system should estimate different dimensions of learner state.

8.1 Mastery estimate
General competence on the concept.

8.2 Memory strength
How reliably it can be retrieved.

8.3 Recognition strength
Can the learner recognize it even if they cannot produce it?

8.4 Reconstruction strength
Can they rebuild it from clues?

8.5 Reasoning strength
Can they explain why and derive consequences?

8.6 Transfer strength
Can they use it in disguised contexts?

8.7 Pressure tolerance
Can they still perform under time pressure?

8.8 Misconception risk
Which false beliefs are most likely active?

8.9 Retention decay rate
How quickly this concept slips away.

8.10 Confidence calibration
Do they know when they know, or do they overguess?

Now the engine can make better decisions.

Examples:

If memory_strength is low:

use recall, recognition, reconstruction, recovery

If misconception_risk for “osmosis vs diffusion” is high:

use compare-and-contrast, misconception trap, classification

If mastery is decent but transfer_strength is weak:

use disguised application and representation conversion

If pressure_tolerance is weak:

gradually introduce timed retrieval

If concept is decaying:

issue delayed resurfacing and recovery ladders

9. End-to-end generation flow

Step 1 — Session goal arrives

Step 2 — Target concept selection

Step 3 — Learner state lookup
The engine checks StudentConceptState.

Step 4 — Intent selection
The system decides what truth it wants to reveal now.

Example:

“Do they confuse these two concepts?”

“Can they continue a solution correctly?”

“Can they still do this under speed?”

Step 5 — Family selection
The system chooses the best question family.

Step 6 — Primitive extraction
The engine fetches the relevant concept building blocks.

Step 7 — Transformation
The engine applies operators to create the internal challenge structure.

Step 8 — Variant composition
The engine adjusts freshness, disguise, cues, and difficulty.

Step 9 — Surface rendering
The question becomes student-facing text, visuals, or timing logic.

Step 10 — Validation
The question is checked before release.

Step 11 — Delivery
The student receives the item in the proper mode.

Step 12 — Response capture
The system records answer, time, confidence, hint use, and error pattern.

Step 13 — Response analysis
The engine infers what the attempt means.

Step 14 — Learner model update
The concept-level state is updated.

Step 15 — Next-step decision
The system decides whether to:

reinforce

deepen

repair

revisit later

increase pressure

move forward

branch into prerequisite repair

10. The branching logic after student response
This is crucial. The engine should not respond to “wrong” with the same generic behavior.

Instead:

If correct and fast
Increase fluency confidence and optionally increase difficulty or move to transfer.

If correct but slow
Knowledge may be present but not fluent.
Queue speed or lighter pressure questions later.

If correct but confidence low
Treat as fragile knowledge.
Schedule confirmation and reinforcement.

If wrong with misconception pattern
Launch targeted misconception repair.

If wrong but close
Possibly conceptual understanding exists but retrieval is weak.
Move to reconstruction or cue-supported recovery.

If wrong at first-step stage
The learner likely cannot initiate the process.
Repair entry strategy, not just whole-problem practice.

If right on recognition but wrong on recall
The learner knows the answer when shown it but cannot produce it independently.
That means memory not yet stable.

If right on recall but wrong on transfer
Knowledge is memorized but not deeply understood.

If right untimed but wrong under pressure
Pressure tolerance, not raw knowledge, is the issue.

This is where the engine begins to feel alive.

12. The validator rules you absolutely need
Before a question goes live, validate:

Semantic correctness
The question actually matches the concept truth.

Diagnostic integrity
If a distractor is meant to represent a misconception, it must truly map to that misconception.

Ambiguity check
There must not be two equally acceptable answers unless the question is built for open-ended scoring.

Level appropriateness
Language and concept load must match learner level.

Curriculum alignment
The generated question must stay within what the syllabus expects.

Solution traceability
There must be a valid explanation path and answer justification available.

Duplicate protection
Do not serve near-duplicates too often.

Mode suitability
A long reasoning prompt should not accidentally enter rapid-fire Mental Mode.

Here is the hidden pipeline in simple form:

def request_question(student_id, mode, concept_pool):
    target_concept = select_target_concept(student_id, mode, concept_pool)
    learner_state = load_student_concept_state(student_id, target_concept)

    evaluative_intent = choose_intent(mode, learner_state, target_concept)
    question_family = resolve_family(evaluative_intent, target_concept, learner_state)

    primitives = extract_primitives(target_concept, question_family)
    variant_plan = compose_variant(learner_state, mode, question_family, target_concept)

    internal_question = transform(primitives, question_family, variant_plan)
    surfaced_question = render_surface(internal_question, mode, learner_state)

    validated_question = validate_question(surfaced_question)
    save_generated_question(validated_question)

    return validated_question
And after answering:

def process_attempt(student_id, generated_question_id, response):
    question = load_generated_question(generated_question_id)
    analysis = analyze_response(question, response)

    update_attempt_record(student_id, generated_question_id, analysis)
    update_student_concept_state(student_id, question.concept_id, analysis)

    next_action = decide_next_action(student_id, question.concept_id, analysis)
    return next_action
16. The hidden philosophy that will make this engine great
The strongest principle is this:

A question is not content.
A question is an instrument for extracting evidence.

That means your factory should always ask:

what evidence of learning do I want?

what kind of mental move will reveal that?

what form of question will force that move?

what will the wrong answer tell me?

what should happen next depending on that answer?

Once the factory thinks like this, it stops being a random generator and becomes a real educational intelligence system.

10. Generation flow in contract form
Now let us state the exact runtime pipeline more formally.

Step A — receive request
A mode sends QuestionRequest.

Example:

student_id = S1

mode = Memory

concept_scope = [fractions, decimals]

preferred_formats = [MultipleChoice, ShortAnswer]

Step B — target concept resolver
The factory determines what concept to work on using:

forced concept if present

review queue if due

weakest concept if mode is Knowledge Gap

decaying concept if mode is Memory

pressure-ready concept if mode is Mental

high-mastery stretch concept if mode is Elite

This should be a separate resolver, not mixed into generation logic.

Step C — learner state fetch
Load StudentConceptState.

If missing, initialize with neutral defaults.

Step D — intent selection
Choose the evaluative question:

verify direct recall

test concept boundary

expose misconception

test reasoning depth

test next-step competence

test speed under pressure

confirm retention after delay

Step E — family resolution
Map intent to a usable question family.

This should consider:

does this concept have a solution graph?

does it have misconception records?

does it have enough examples?

does the mode allow open response?

Step F — material extraction
Gather a QuestionMaterialBundle.

Step G — difficulty and freshness planning
Compute:

difficulty profile

format selection

representation choice

variant signature

cue level

distractor strength

time pressure

Step H — transformation
The transformation engine creates an internal draft.

Example:

family = ErrorDetection

use solution graph

choose common invalid sign step

create correct path with one corruption

attach misconception mapping

Step I — surface rendering
Turn it into displayed text/options/payload.

Step J — validation
Reject or repair if:

broken options

ambiguous key

poor fit for age level

duplicate similarity too high

missing explanation

Step K — persistence
Persist generated_questions.

Step L — return to mode engine
The mode engine receives a clean GeneratedQuestion.

12. Scoring and learner-state math
Do not overcomplicate at first, but do not make it crude.

A strong first version can use weighted updates.

12.1 Mastery update
Example internal logic:

new_mastery =
old_mastery * 0.75
+ correctness * 0.15
+ reasoning_quality * 0.05
+ transfer_success * 0.05
12.2 Memory strength update
Use:

recall correctness

latency

delayed success

cue dependence

Example:

correct direct recall fast → strong positive

correct only with recognition → small positive

correct only with clues → reconstruction stronger than memory

wrong after long delay → decay increase

12.3 Misconception risk update
If a wrong option consistently maps to the same misconception, raise its risk score.

misconception_risk[m] += weighted evidence
If learner later answers repair questions correctly, gradually reduce it.

12.4 Pressure tolerance update
Compare performance under timed vs untimed conditions.

If learner collapses only under time, do not lower concept mastery too harshly.
Lower pressure tolerance instead.

This distinction is very important.

reduce misconception risk

If correct answer but weak explanation:

mark shallow mastery

keep transfer and reasoning strengths lower

If explanation includes misconception:

sharply raise misconception risk

trigger repair sequence even if final answer looked superficially right

4.9 Common failure cases
scoring becomes too subjective

language ability confuses concept ability

prompt too broad

open response too heavy for Mental Mode or young learner

4.10 Mode usage
Knowledge Gap Mode: strong

DNA: very strong

Journey Mode: strong

Elite Mode: strong

Memory Mode: secondary

Mental Mode: generally limited

5. Application Family Spec
5.1 Purpose
To test whether the learner can use a principle or concept in a concrete situation.

5.2 When to use
Use Application when:

concept understanding has been established

system wants proof of usable knowledge

learner seems to know theory but not practice

preparing for exam-style scenario questions

5.3 Required inputs
target concept

application contexts

mapping rule from concept to situation

optional distractor misconceptions

optional scenario representations

5.4 Generation logic
Core transformation rule
Embed the concept in a case and require the learner to apply the correct idea.

Generator steps
choose concept

choose scenario type:

textbook familiar

semi-real-world

strongly real-world

hide or reveal concept name depending on difficulty

inject irrelevant details if desired

render question

Example
Concept: density
Question:

“Why does oil float on water after a spill?”

Concept: evaporation
Question:

“Why do people spread grains out in the sun rather than heap them in a pile?”

pub struct ApplicationQuestionPayload {
    pub scenario_id: Option<String>,
    pub target_concept_id: String,
    pub concept_visibility: ConceptVisibility,
    pub irrelevant_detail_load: f32,
    pub expected_mapping_rule: String,
}
5.6 Scoring logic
correctness

if open-ended, reasoning link between concept and situation

whether learner chose right principle or wrong competing principle

partial credit if principle recognized but explanation weak

5.7 Validator rules
scenario must truly instantiate the concept

irrelevant details must not obscure unfairly

alternative concepts must not fit equally well

if multiple correct approaches exist, scoring must support them

5.8 Learner-state update effects
If correct:

increase application_strength or mastery_estimate

stronger positive than simple recognition

modest transfer readiness boost

If wrong but recall still good:

indicates inert knowledge

lower application-related mastery but not necessarily memory strength

5.9 Common failure cases
scenario too vague

concept too explicitly named, making it shallow

application question accidentally becomes recall

real-world context introduces unrelated bias

Knowledge Gap Mode: useful after foundation

6. Error Detection Family Spec
6.1 Purpose
To test whether the learner can identify an incorrect step, invalid claim, or broken reasoning path.

6.2 When to use
Use Error Detection when:

diagnosing misconceptions

checking process awareness

sharpening precision

testing whether learner understands not just answer but method

6.3 Required inputs
valid solution path or valid reasoning chain

one injectable invalid step

error type classification

optionally misconception mapping

correction anchor

6.4 Generation logic
Core transformation rule
Generate a correct worked path, then deliberately corrupt one point.

Generator steps
choose target concept/process

fetch valid solution graph or reasoning chain

choose one invalid transition

inject fault

choose question frame:

“Which step is wrong?”

“What is the mistake here?”

“Why is this reasoning incorrect?”

render item

Example
a math solution with one sign error

a science explanation that confuses evaporation with boiling

a grammar rule applied wrongly to a sentence

}
6.6 Scoring logic
Possible sub-scores:

identified wrong step

named why wrong

optionally proposed correct alternative

Can be split:

detection score

explanation-of-error score

6.7 Validator rules
only one intended fault unless multi-fault explicitly designed

surrounding steps must remain coherent

wrongness must be real, not debatable

correction anchor must exist

if step numbering shown, target step must be unambiguous

6.8 Learner-state update effects
If learner spots error and explains it:

increase reasoning_strength

reduce misconception risk for that fault

increase precision-related mastery

If learner misses specific error:

raise linked misconception risk

possible need for correction family next

6.9 Common failure cases
injected error makes later steps impossible in confusing ways

multiple accidental errors

learner spots wrong step but system cannot capture why

poor solution graph causes low-quality items

6.10 Mode usage

Knowledge Gap Mode: very strong

Journey Mode: strong

Elite Mode: strong

Memory Mode: limited

Mental Mode: only in short, sharp versions

7. Misconception Exposure Family Spec
7.1 Purpose
To reveal whether a specific false belief is active in the learner’s mind.

7.2 When to use
Use this family when:

concept has known confusion patterns

engine needs precise diagnosis

learner repeatedly fails in patterned ways

system wants to separate ignorance from false understanding

7.3 Required inputs
misconception record

trigger scenario

trap response options or trap prompt

correction logic

optionally contrast concept

7.4 Generation logic
Core transformation rule
Create a context where the misconception feels tempting, then observe whether learner falls for it.

Generator steps
select target misconception

fetch trigger pattern from misconception record

choose item form:

direct statement judgment

scenario trap

near-neighbor multiple choice

compare/contrast trap

render item so misconception is attractive but not unfair

Example
Misconception:

“Evaporation only happens when liquid is hot.”

Generated item:

“A puddle outside disappears during cool weather. Which process best explains this?”

Wrong attractive option:

“The water boiled away.”

7.5 Output structure
pub struct MisconceptionExposurePayload {
    pub misconception_id: String,
    pub trigger_pattern_id: Option<String>,
    pub trap_strength: f32,
    pub expected_false_choice_ids: Vec<String>,
    pub repair_sequence_ref: Option<String>,
}
7.6 Scoring logic
correct / incorrect

if wrong, whether wrong answer maps exactly to target misconception

misconception evidence confidence

This family is more about diagnosis than raw marks.

7.7 Validator rules
misconception must be real and documented

trap must not rely on trick wording alone

wrong answer must be plausible specifically because of misconception, not random confusion

correct answer must still be reasonably accessible

7.8 Learner-state update effects
If learner falls for the trap:

increase misconception_risk[target] strongly

queue repair immediately or soon

possibly do not lower general mastery too harshly if error is highly localized

If learner avoids trap consistently:

gradually reduce misconception risk

increase boundary clarity estimate

7.9 Common failure cases
trap becomes unfair trick

wrong option attracts due to reading issue, not misconception

same misconception repeatedly probed without repair

7.10 Mode usage
DNA: primary

Knowledge Gap Mode: primary

Journey Mode: strong

Elite Mode: good for subtle boundary sharpening

Memory Mode: limited

Mental Mode: possible in fast boundary drills

8. First-Step Family Spec
8.1 Purpose
To test whether the learner knows how to begin solving a problem or analyzing a task.

8.2 When to use
Use this family when:

learner freezes at start

learner often chooses wrong method

full problem solving is too heavy for diagnosis

system wants to isolate strategic entry knowledge

8.3 Required inputs
full solvable problem

solution graph

valid initial transition(s)

invalid common opening moves

8.4 Generation logic
Core transformation rule
Present the problem in full, but ask only for the best first move.

Generator steps
choose process-based concept

fetch solution graph

identify initial valid node(s)

collect common wrong first moves

choose format:

multiple choice

short response

step selection

render item

Example
Math:

“What should you do first to solve 3x + 5 = 17?”

Science:

“What should be checked first when classifying this transport process?”

9. Next-Step Family Spec
9.1 Purpose
To test whether the learner can continue a partially completed reasoning or solution process correctly.

9.2 When to use

learner can start but breaks mid-process

process diagnosis is needed

system wants to scaffold full problem solving

stepwise mastery is being trained

9.3 Required inputs
partial state from solution graph

valid next transitions

common invalid transitions

optionally error-tagged branches

9.4 Generation logic
Core transformation rule
Freeze the process at a defined state and ask what should happen next.

Generator steps

extract valid next move(s)

retrieve common invalid next move(s)

render item

10. Retention Check Family Spec
10.1 Purpose
To verify whether previously learned knowledge remains accessible after time has passed.

10.2 When to use
Use this family when:

concept has not been seen recently

system suspects decay

spaced review is due

memory strength needs confirmation

10.3 Required inputs
previously encountered concept

prior attempt history

delayed review schedule

alternate wording or alternate representation

10.4 Generation logic
Core transformation rule
Re-ask an already learned concept after delay, preferably in changed surface form.

Generator steps
choose due concept from review queue or decay model

inspect last seen form

choose different surface form or phrasing

select appropriate cognitive type:

recall

reconstruction

lightweight application

render delayed check

Important:
A retention check is not defined by its surface form alone.
It is defined by the fact that it is asked after meaningful delay.

10.6 Scoring logic
correctness

response time relative to concept history

support dependence

retention stability against prior history

A learner getting it right after 10 days in changed form is much stronger evidence than getting it right immediately after lesson.

10.7 Validator rules
concept must have meaningful prior exposure

variation should not be identical repetition unless intentional

difficulty shift should be controlled

delayed check should still reflect same concept evidence

10.8 Learner-state update effects
If correct after delay:

increase memory_strength strongly

reduce retention_decay_rate

strengthen stable mastery estimate

If wrong after delay:

increase decay estimate

queue recovery sequence

do not treat it the same as first-time ignorance

10.9 Common failure cases
asking too soon and calling it retention

repeating same wording exactly

using a much harder representation and misattributing failure to decay

11. Recovery Family Spec
11.1 Purpose
To rebuild access to weakening or partially lost knowledge.

11.2 When to use
Use Recovery when:

retention check failed

memory appears fragmentary

learner once knew concept but now struggles

system wants to repair rather than restart from zero

11.3 Required inputs
weakened concept

clue ladder

prerequisite anchors

target primitive

support gradient

prior failure history

11.4 Generation logic
Core transformation rule
Start with supported access and gradually reduce support as learner re-approaches the concept.

Generator steps
choose weakened concept

estimate support level needed

build clue ladder:

strong cue

moderate cue

light cue

direct retrieval

choose current recovery rung

render question

This can be single-item or mini-sequence based.

correctness

progression across rungs

12. Retrieval Under Pressure Family Spec
12.1 Purpose
To test recall, recognition, or quick reasoning under compressed time conditions.

12.2 When to use
Use this family when:

system wants fluency and sharpness

concept is already at least moderately known

learner is in Mental Mode

readiness under pressure matters

12.3 Required inputs
short-response-capable concept material

timing profile

burst length

streak logic

pressure escalation plan

12.4 Generation logic
Core transformation rule
Take short cognitive items and deliver them with reduced response windows, often in chained form.

Generator steps
choose pressure-suitable concept items

select question micro-family:

rapid recall

rapid recognition

rapid compare/contrast

rapid classification

assign timer

assign streak/chain behavior

define penalty/reward behavior

render burst sequence

Important
Pressure questions must be short in reading load.
Otherwise you are measuring reading speed more than concept sharpness.

correctness

13. Cross-family engineering note
Each family should expose a common contract like this:

pub trait QuestionFamilyGenerator {
    fn generate(
        &self,
        request: &FamilyGenerationRequest,
        materials: &QuestionMaterialBundle,
        learner_state: &StudentConceptState,
    ) -> Result<InternalQuestionDraft, FamilyGenerationError>;

    fn validate(
        &self,
        draft: &InternalQuestionDraft,
    ) -> Result<(), FamilyValidationError>;

    fn analyze_attempt(
        &self,
        question: &GeneratedQuestion,
        submission: &AttemptSubmission,
    ) -> Result<FamilyAttemptAnalysis, FamilyAnalysisError>;
}
That is very important.

Because then each family owns:

how it is generated

how it is validated

how its answers are interpreted

instead of forcing one giant generic engine to do everything badly.

14. Recommended first implementation order
Do not implement these 12 families randomly.

Build in this order:

Wave 1
Recall

Recognition

Application

Error Detection

These give immediate practical value.

Wave 2
Reconstruction

Explanation

Misconception Exposure

Retention Check

These make the engine smarter.

Wave 3
First-Step

Next-Step

Recovery

Retrieval Under Pressure

These make your product distinctive.

15. The hidden rule that must guide every family
For every family, your engineers should ask:

What mental act is being forced?

What evidence does success provide?

What does each wrong answer mean?

What state dimensions should change after this?

Which mode benefits from this family?

What makes a generated version valid?

If those answers are clear, the family is build-ready.

The next best step is to define the shared scoring model and learner-state update framework so all these families feed one consistent adaptive intelligence system instead of each one updating state differently.
