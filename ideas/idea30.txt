3. The most important distinction: raw hours vs effective hours
The 100 hours should not be plain minutes on a clock.

You want:

Raw Time
Actual elapsed study time.

Effective Learning Credit
How much real progress was made.

Verified Retained Progress
How much stuck after spacing.

This means a session outcome must produce a weighted credit, not a flat time addition.

A simple model:

Effective Credit = Raw Minutes × Focus Factor × Task Quality Factor × Performance Factor × Retention Confidence Factor

Example:

60 minutes

focus factor = 0.9

task quality = 1.0

performance = 0.8

retention confidence = 0.7

Effective credit = 60 × 0.9 × 1.0 × 0.8 × 0.7 = 30.24 “effective minutes”

This prevents fake progress.

A weak distracted session should not count like a sharp session.

4. Capacity model
Now let us define how the coach decides how much usable time exists before the exam.

A. Raw capacity
Sum of all available windows until exam.

B. Realistic capacity
Raw capacity adjusted by real-world behavior.

Formula concept:

Realistic Capacity = Raw Capacity × Attendance Reliability × Fatigue Adjustment × Time-of-Day Suitability × Schedule Friction Adjustment

Where:

attendance reliability comes from actual history

fatigue adjustment reduces over-optimistic planning

time-of-day suitability reflects when the student performs well

schedule friction accounts for overly fragmented or unrealistic slots

Example:
Raw remaining capacity = 140 hours
Attendance reliability = 0.82
Fatigue adjustment = 0.9
Time-of-day suitability = 0.95
Friction adjustment = 0.92

Realistic capacity = 140 × 0.82 × 0.9 × 0.95 × 0.92 ≈ 90 hours

This tells the truth.

If the coach needs 100 effective hours but realistic capacity is only 90, it must adapt immediately.

6. Workload distribution formula
Now we define how to spread the 100 hours.

Let:

H_rem = effective hours remaining

C_day[i] = weighted realistic capacity of each day until exam

W_total = sum of all C_day[i]

Target_day[i] = share of work assigned to day i

Then:

Target_day[i] = H_rem × (C_day[i] / W_total)

That gives a base spread.

But then you must modify it with pedagogic logic:

not too many hard sessions back to back

deliberate gaps for decay testing

heavier weekends

lighter recovery days

increased review density near exam

topic-specific revisit timing

So this is only the base allocation layer.

Pedagogically sensitive miss
Dangerous because it breaks spacing or a prerequisite chain.

Pace-sensitive miss
Dangerous because plan is already tight.

Critical miss
Dangerous because exam is near and the topic was high urgency.

Miss response logic
Benign miss
absorb using buffer

no student stress

re-evaluate next session only

Pedagogically sensitive miss
create replacement within the acceptable spacing window

protect follow-up session from becoming invalid

Pace-sensitive miss
split debt across next 2–4 suitable windows

reduce optional low-value work

Critical miss
immediate replanning

possible schedule warning

possible shift from mastery mode to high-yield exam mode

Missed debt formula concept
Debt Severity =
session importance × timing sensitivity × pace pressure × exam proximity

Higher debt severity means faster recovery insertion.

13. Buffer logic
Buffer is essential. Without it, the system will break.

You want three kinds of buffer.

A. Time buffer
Reserved empty capacity across the plan.

B. Cognitive buffer
Days or half-days that are intentionally lighter.

C. Schedule buffer
Windows where no specific heavy session is assigned yet, but they are held open for future adjustment.

Recommended behavior
In normal mode:

reserve 10–20% of total realistic capacity as buffer

In tight mode:

reserve 5–10%

In critical mode:

minimal formal buffer, but still preserve some recovery logic or burnout risk will rise

Buffer usage rules
Use buffer for:

missed session recovery

extra review before exam

unexpected weakness emergence

spillover from difficult sessions

exam simulation correction

Do not waste buffer on:

random low-yield repetition

generic filler review

repeated safe-topic drilling just because it feels productive

14. Planned spacing logic
Now let us define how the engine decides when to revisit a taught topic.

Each topic needs a dynamic spacing interval.

Base intervals by stability
introduced: revisit in 1 day

partial: revisit in 2–3 days

fragile: revisit in 1–2 days

usable: revisit in 4–6 days

stable: revisit in 7–12 days

exam-ready: revisit strategically based on exam distance and importance

But spacing should be modified by:
confusion risk

student forgetting profile

topic complexity

exam importance

recent errors

whether learning was shallow or deep

whether understanding was demonstrated or merely recognized

So the next review date is not fixed. It is computed.

Example:
A topic is “usable,” so normally 5-day interval.
But:

confusion risk high

student forgets quickly

exam frequency high

The engine shortens interval to 3 days.

15. Anti-repetition rules
A coach must not become boring.

So the engine needs explicit repetition guards.

Repetition penalties
A session gets penalized if:

same topic type repeated too many times consecutively

same session format repeated too often

same question style repeated with no new value

same cognitive demand repeated while another neglected queue exists

This should affect ranking.

You want:

variety with purpose

not randomness

not monotony

So two similar sessions close together must have a reason:

reinforcement

correction

pressure shift

decay check

transfer practice

If not, penalize.

16. Auto-trigger decision logic
For auto mode, the system should not trigger merely because time exists.

It should trigger because conditions are satisfied.

Auto-trigger rules
A session can auto-trigger if:

it is inside an approved availability window

student did not already complete a meaningful equivalent session

no hard block is active

session urgency above threshold

student’s recent overload not above threshold

no stronger pending session has displaced it

Smart trigger timing
Inside a given window, the coach should choose the best moment based on:

historical start behavior

completion success by time

focus quality by hour

lateness patterns

current schedule pressure

So if the student says they are available 6–9 pm, the system may learn:

6 pm starts often fail

7:15 pm starts succeed more

8:30 pm starts are too rushed

Then it targets 7:15.

17. Hybrid mode logic
Hybrid should likely be your default.

Example policy:

bonus/free-now sessions: manual

exam simulation sessions: pre-confirmed auto

This gives structure without becoming irritating.

A. Presence state

B. Progress state

quiet but genuinely learning

Session lifecycle states
1. Scheduled
The coach has planned the session, but the student has not started.

2. Launched but not started

The student is clearly engaging:

answering

typing

navigating

annotating

4. Active passive
The student is consuming learning material in a valid passive way:

reading a lesson

The student is quiet, but the silence is expected:

thinking through a problem

planning an essay

solving on paper

mentally processing a diagram

working through steps off-screen

There has been no interaction for a while, but it is too early to call it idle.

The system thinks engagement may have stopped.

The student has not responded within the quiet window and is now officially idle.

A system event occurred:

app backgrounded

screen locked

call came in

battery died

app crashed

device restarted

10. Manually paused
The student explicitly paused.

11. Manually stopped early
The student intentionally ended before session completion.

The student left and did not return within the resume window.

13. Completed
The core session objective was achieved.

14. Reactivated
The student returned and resumed the same coaching episode.

3. When the time clock should start and stop
You said this well already: time should be calculated from the first click or input to the last click or input.

That is right as a base rule, but it needs refinement.

The correct rule
The session clock should start at the first meaningful learning interaction, not merely when the app opens.

So these should not start the clock:

auto-launch to home screen

notification opened but no session interaction

app foregrounded without student action

These should start the clock:

tap “Start Session”

first answer input

first navigation inside the session

first playback start on assigned material

first voice response

first scratchpad/note action

first annotation or problem interaction

You should store multiple time metrics
Do not store only one “time spent” number.

You need at least these:

1. Gross episode span
From first meaningful interaction to final meaningful interaction within that coaching episode.

2. Active engagement time
Time supported by strong interaction evidence.

3. Valid passive learning time
Time spent reading, watching, or listening in a way the system considers legitimate.

4. Thinking / off-screen solving time
Counted only when context supports it.

5. Idle time
Time from idle start until return or session park.

6. Interrupted/background time
Time lost to calls, lock screen, app switching, crash, etc.

7. Counted study time
This is the number the coach should mostly trust:

Counted study time = active engagement + validated passive time + approved thinking/off-paper time

Because if someone opens the app for 50 minutes but was truly engaged for 18, the system must not tell itself they studied for 50.

last_meaningful_event_at

idle_started_at

idle_confirmed_at

Idle does not begin when the timeout fires
It begins at the last meaningful interaction.

Example:

last answer input: 7:42:10

idle threshold crossed at 7:44:10

idle confirmed at 7:45:00

The idle start time should be:
7:42:10, not 7:45:00

That matters for accurate analytics.

A single idle threshold across the whole app will fail.

The coach should decide idleness based on the kind of activity happening.

The student may be silent for a long time and still be working.

estimated reading time

complexity

student’s reading pace

whether the content is new or review

A short paragraph and a dense page should not share the same threshold.

B. Worked example mode
The student may pause to inspect steps.

Allowed silence should be moderate.

C. Solving / question mode
Silence may mean:

thinking

solving on paper

stuck

gone

This mode needs stronger intelligence.

The coach should look at:

expected solve time

question difficulty

student’s personal median response time

whether paper-solving is likely

whether prior items showed long thinking windows

D. Video / audio mode
If assigned media is actively playing, the student may still be learning without touching the screen.

So this should count as passive engagement, but not indefinitely.

The system should still watch for:

playback progress

pause events

repeated rewinds

audio completion

app background behavior

headphone disconnect if relevant and allowed by OS

E. Essay / long response mode
Long silent intervals are normal if the student is planning or drafting.

Typing inactivity alone is not enough to declare abandonment.

F. Timed quiz / mock mode
This is different.

In strict exam simulation:

backgrounding may be treated as a rule event

silence is part of the exam experience

In coached timed mode:

the system can intervene more gently

So timed modes must have their own policy.

6. The coach needs a “thinking is not abandonment” layer
This is one of the most important pieces.

A student can be deeply engaged without touching the screen.

Examples:

doing a long calculation on paper

sketching a biology diagram

writing an essay outline in a notebook

thinking through a proof

reading a dense worked solution

mentally separating two confusing concepts

If the coach marks this as abandonment, it becomes stupid.

This state can be entered either:

explicitly, through a button like “Working it out” or “Solving on paper”

or implicitly, because the current task type supports long quiet thinking

When in this state:

the idle threshold is extended

the coach may show a very light prompt after a while

the time may still count, but with controlled confidence

if the silence becomes too long, the system moves to suspected idle, not immediately abandoned

This is another major distinction.

A student may still be touching the screen but not progressing.

Examples:

repeated wrong attempts

toggling options with no submission

repeated hint opening

excessive dwell time on one question

small interactions but no forward movement

answer changes over and over

scrolling up and down the same explanation repeatedly

This is not abandonment.

This is struggle or stall.

So you need a separate progress-state model:

advancing

slowed

stalled

confused

looping

recovered

completed

If the student is active but stalled, the coach should intervene:

offer a hint

break the item into steps

switch to a worked example

ask a smaller question

suggest a short reset

That is much better than waiting for abandonment.

7. Threshold framework
The engine needs three time thresholds at minimum.

T1. Quiet threshold
The first point where the engine says:
“Silence is becoming noteworthy.”

T2. Idle confirmation threshold
The point where the engine says:
“This is now real idleness unless disproven.”

T3. Park / abandonment threshold
The point where the current active segment is closed and the episode is parked or closed.

These must be dynamic, not fixed app-wide.

8. Dynamic threshold formula
Each mode gets a threshold profile.

A simple structure:

Quiet Threshold = Base Mode Value × Difficulty Multiplier × Student Pace Multiplier × Context Modifier

Idle Threshold = Quiet Threshold + Secondary Buffer

Park Threshold = Idle Threshold + Recovery Buffer

Where:

Base Mode Value depends on current activity type

Difficulty Multiplier grows for harder items

Student Pace Multiplier reflects that learner’s typical tempo

Context Modifier adjusts for paper-solving likelihood, essay planning, passive media, etc.

9. Recommended baseline thresholds by mode

Strict mock exam

10. Reading mode logic
Reading mode must not punish careful students.

Use content-aware timing.

Let:

W = visible word count in current chunk

R_user = student estimated reading speed

C = complexity multiplier

Then:
Expected Read Time = (W / R_user) × C

Possible default:

R_user starts from grade/age heuristic

later personalized using actual reading behavior

C higher for dense science/math explanations than plain text

Then:

quiet threshold can be a fraction of expected read time

idle threshold can be expected read time plus buffer

This is far better than “2 minutes for everybody.”

11. Problem-solving mode logic
This mode is most important because silence is ambiguous.

Let:

ERT = expected response time for the item

D = difficulty modifier

P = user pace profile

O = off-screen solving likelihood

Then:
Quiet Threshold ≈ ERT × P × D
capped within reasonable min/max bounds

Idle Confirmation ≈ Quiet Threshold + buffer
with buffer larger if off-screen solving is likely

Example:

easy mental item: short quiet threshold

geometry construction or algebraic derivation: much longer quiet tolerance

Also:
If the student has explicitly entered “solving on paper” mode, extend the thresholds further.

12. Passive media logic
Passive learning should count, but carefully.

During valid playback
Presence state = active_passive

This counts as valid learning only if:

assigned media is playing

playback position is advancing

speed is within allowed range

not looping suspiciously without progression

not left running forever after meaningful content ends

not backgrounded in disallowed strict modes

Possible counted-time confidence:

medium, not maximum

Repeated unattended playback risk
If the same student repeatedly lets videos run untouched and performs poorly on recall:

passive confidence should be down-weighted

scheduler should reduce trust in passive study time

This is crucial.

From launched_unengaged

From active_interactive

on app background/lock/call → background_interrupted

on completion criteria met → completed

From active_passive
on valid playback progression → remain active_passive

on playback pause/end without new interaction → quiet_grace

on manual input → may shift to active_interactive

on app background in allowed flexible mode → either remain passive-valid for a bounded window or move to interrupted depending on platform rules

on suspicious no-progression media state → idle_suspected

From thinking_solving
on meaningful answer/note/hint/navigation → active_interactive

on silence beyond extended quiet threshold → idle_suspected

on manual pause → manual_pause

on background/lock → background_interrupted

From quiet_grace
on meaningful event → previous active state restored

on threshold escalation → idle_suspected

on background/lock → background_interrupted

From idle_suspected
on meaningful event quickly → reactivating then active state

on no response by idle confirmation → idle_confirmed

on manual pause/stop → those states accordingly

on app background → background_interrupted

From idle_confirmed
on quick return within resume window → reactivating

on no return until park threshold → parked_recoverable

on explicit stop → parked_recoverable or abandoned_closed depending on context

From background_interrupted
on return within interruption resume window → reactivating

on long absence → parked_recoverable

in strict mock mode, background may mark breach_flagged while timer continues or attempt invalidates per rules

From manual_pause
on resume → reactivating

on long elapsed pause → parked_recoverable

on stop → parked_recoverable

From reactivating
short gap → restore exact state

medium gap → restore with recap

long gap → resume by smaller restart flow or split continuation

if the underlying plan expired → convert to new recovery episode

At t_last + T1

17. Counted-time formula
This is critical.

You need multiple time buckets.

Let:

A = active_interactive seconds

P = active_passive seconds

T = thinking_solving seconds

I = idle seconds

B = background/interrupted seconds

M = manual_pause seconds

Then define:
Counted Study Time = A + (P × p_weight) + (T × t_weight)

Where:

p_weight may start around 0.6–0.9 depending on content and evidence

t_weight may start around 0.5–0.85 depending on context and confidence

Never count:

confirmed idle

background interruption

manual pause

unattended open-screen time

You can also store:
High-confidence counted time = A
Expanded counted time = A + weighted passive + weighted thinking

That gives the mastery engine options.
