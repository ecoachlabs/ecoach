1.3 Matching stages

Example:

Stage B: Number-based matching
This is the strongest first-pass signal.

Exact number match
If:

answer number = 4

then assign a strong base score.

Subpart match
If:

question = 3(b)

answer = 3(b)

Parent-child match
If:

question is 3(a), 3(b), 3(c)

answer only says 3

then allow structured expansion logic.

Meaning:

Stage C: Sequence matching
If numbering is weak or missing, rely on order.

Example:

first extracted answer block after Section B probably belongs to first Section B question

second block probably belongs to next question

This should only apply within a section and with lower confidence than explicit numbering.

Stage D: Spatial grouping
If page/image layout is available, use geometry.

Examples:

answer written directly beneath question on same sheet

teacher marks appear beside a block

subparts are clustered spatially

Stage E: Semantic relevance matching
If the answer text contains terms strongly associated with a question, use that as supporting evidence.

Example:
Question:

Answer contains:
“chlorophyll,” “sunlight,” “carbon dioxide,” “glucose”

That boosts alignment.

This should not dominate numbering, but it is useful when numbering is poor.

Stage F: Mark-anchor matching
If teacher marks or comments are adjacent to a known answer number, use that as an anchor.

Example:

red “2/5” beside a block labeled 5

comment “incomplete explanation” near a paragraph under question 6

This strengthens answer-to-question confidence.

Stage G: Bundle metadata boost
If the uploader said:

these are the question pages

these are answer pages

this is Section B

this is a marked script

apply bias in favor of those interpretations.

2. Matching score model
Each possible question–answer pair should receive a score.

Not binary.
Scored.

A good model is a weighted score composed of several features.

2.1 Features
For candidate pair (q, a) compute:

section_match_score

number_match_score

subpart_match_score

sequence_proximity_score

page_proximity_score

semantic_similarity_score

teacher_mark_anchor_score

layout_contiguity_score

bundle_metadata_score

Then compute:

pair_score = weighted_sum(features)

You do not need ML first.
A deterministic weighted engine is enough to start.

2.2 Example weight profile
For early implementation, something like:

number match: 0.35

subpart match: 0.20

section match: 0.15

sequence proximity: 0.10

semantic similarity: 0.08

teacher mark anchor: 0.05

layout contiguity: 0.04

bundle metadata: 0.03

These are just starter weights.
You can tune later.

The point is:
explicit numbering should dominate.

2.3 Confidence classification
After scoring candidate pairs, classify them as:

High confidence: top score clearly stronger than alternatives

Medium confidence: top score reasonable but alternatives close

Low confidence: weak signals or ambiguity

Unresolved: no acceptable candidate

Example threshold logic:

high if score >= 0.82 and margin from second candidate >= 0.15

medium if score >= 0.65 and margin >= 0.08

low if score >= 0.50

unresolved otherwise

The exact values can change, but you need:

absolute confidence

relative confidence margin

Both matter.

One question → one answer block containing subparts

1. What this layer is
Call it the:

Daily Adaptive Planner
Its role is to sit beneath the Weekly Planning Generator and constantly tune execution.

The weekly planner says:

what the week should look like

The daily adaptive planner says:

what today should look like now

what should happen next after what just happened

what should be moved, intensified, shortened, delayed, or changed

So the weekly plan is the strategy.
The daily planner is the tactical controller.

2. Core purpose
The Daily Adaptive Planner should do five jobs:

decide what today’s active blocks should be

adapt after each session or event

protect the student from overload or bad sequencing

keep the week on track despite disruption

respond quickly to new evidence

That means it is not just a scheduler.
It is a live replanning engine.

3. Core planning philosophy
This layer must obey one key rule:

Do not rigidly follow the weekly plan if reality has changed.
But do not destroy the weekly plan because of every small change.

That balance matters.

The daily adaptive planner must avoid two extremes:

Wrong extreme 1: rigidity
The plan says Algebra at 6 PM, so it stays Algebra no matter what.

Bad, because:

the child may already have shown strong recovery in Algebra

a new marked science test may have arrived

the child may be mentally too tired for a deep Algebra block

Wrong extreme 2: chaos
Every new signal reshuffles everything.

Bad, because:

the child loses continuity

the system becomes unpredictable

deep progress never compounds

So the planner must be responsive, but controlled.

6. Levels of adaptation
Not every event should cause the same size change.

The planner should work in layers.

Level 1: Micro-adjustment
Small changes within the day.
Examples:

shorten a block

swap order of two blocks

replace deep session with lighter review

add quick recall after strong performance

Level 2: Daily rebalancing
Moderate same-day changes.
Examples:

move a block to tomorrow

insert retest

replace scheduled topic with more urgent one

split a heavy block into two smaller ones

Level 3: Weekly rebalancing
Larger changes across the remaining week.
Examples:

intensify a topic after repeated failure

downgrade a topic after improvement

rebuild the next three days because new evidence arrived

This tiering prevents overreaction.

Before each block

After each block

8. Start-of-day planner behavior
At the beginning of each day, the planner should produce a Today Plan, not just expose the week.

The Today Plan should answer:

which one is non-negotiable

which one can be skipped if time collapses

10. Reactions to session outcomes
Now define what happens after each type.

10.1 Strong success
Reaction:

keep schedule unless the result changes priorities

consider moving topic slightly toward maintenance

possibly unlock a slightly harder follow-up

do not waste another deep block immediately unless the topic is still high urgency

Example:
Student performs strongly in Algebra repair.
Then:

keep quick retest later

reduce extra reteach

protect time for other weak areas

10.2 Qualified success
Reaction:

continue with planned next step

keep retest

do not downgrade yet

maybe shorten next reinforcement instead of removing it

10.3 Weak success
Reaction:

schedule reinforcement soon

do not assume mastery

maybe keep tomorrow’s follow-up intact

avoid promoting topic too early

10.4 Partial completion
Reaction depends on why.

Time shortage
split remainder into smaller follow-up block

Fatigue
move remainder to lighter time/day

Confusion
insert simpler scaffold before continuing

This is important:
the system should diagnose why the partial completion happened.

10.5 Failure
Reaction:

do not blindly continue harder work

diagnose failure type

possibly replace next block

insert recovery block

escalate if repeated

Examples:

if concept failure → reteach

if fatigue failure → lighten load

if timing failure → remove pressure temporarily

10.6 Skipped
Reaction:

do not automatically pile backlog

classify skipped block:

critical

important

optional

Critical blocks should be rescheduled.
Optional blocks may be dropped.
Important blocks may be merged later.

That is how the system stays realistic.

11. Fatigue and cognitive load handling
This is a major differentiator.

The planner must understand that a student’s academic ability is not constant across the day.

So the system should estimate:

current mental freshness

frustration

recent heavy-load density

drop in accuracy compared to normal

time-on-task tolerance

This does not need perfect psychological modeling at first.
A good rule-based estimate is enough.

11.1 Fatigue signals
Possible signals include:

slower than usual completion

accuracy drop late in session

excessive hint usage

repeated careless errors

skipping explanation steps

abandonment of blocks

user feedback like “I’m tired” later if supported

11.2 Fatigue reactions
If fatigue is likely:

replace deep block with lighter review

shorten block duration

move hard work earlier tomorrow

use maintenance instead of fresh heavy instruction

preserve only the day’s most critical mission

This is how the coach behaves humanly.

12. Dependency-aware daily changes
Some things should not be moved independently.

Example:

if today’s reteach is skipped, tomorrow’s retest may become invalid

if concept block fails, the follow-up timed drill should be postponed

if answer-structure block fails, essay simulation should be downgraded

So every block should carry dependency tags like:

requires_reteach_done

requires_topic_stabilized

requires_previous_block_completion

This helps the daily planner avoid nonsense sequencing.

13. Daily reprioritization rules
When the daily planner rebalances, it should use a controlled priority ladder.

Highest daily priority candidates
urgent exam-related core weakness

retest due after intervention

new severe evidence from upload

critical dependency blocks

weekly non-negotiable continuity block

Medium priority
standard tactical work

support blocks

medium-severity weakness repairs

Lower priority
optional stretch work

strong-topic reinforcement

nice-to-have exploration

This is the order in which the day should compress when time is limited.

14. New evidence arriving midweek
This is one of the most important use cases.

The system finds:

internal exam is close

this affects current goals

The daily planner should do one of these:

If urgency high
update Thursday/Friday immediately

insert one evidence-response block

maybe reduce lower-value block

If urgency moderate
adjust the next available day

show coach note explaining the change

If confidence low
wait for user confirmation before major plan changes

This gives the coach responsiveness without chaos.

15. Retest-driven adaptation
Retests are powerful because they reveal whether an intervention worked.

The daily planner must use them actively.

15.1 If retest succeeds strongly
reduce immediate further drilling

move topic toward fragile/stable

free time for other priorities

15.2 If retest succeeds weakly
keep one follow-up reinforcement

do not declare recovery yet

15.3 If retest fails
escalate

change intervention type

revisit diagnosis

possibly split topic into subskills

This is where the coach stops repeating the same bad solution.

Instead classify missed work:

Keep

dependency-sensitive

Merge
Blocks that overlap can be combined.

Reduce
Some blocks can be shortened.

Drop
Low-value optional blocks can disappear.

This is essential for realism.

17. Adaptive block replacement
Sometimes a planned block should be replaced with a better one.

Examples:

Replace deep reteach with quick recovery review
if student already showed strong improvement earlier that day.

Replace timed drill with untimed method block
if performance collapse suggests timing is not the real next step.

Replace optional maintenance with urgent upload-response session
if new evidence is serious.

This replacement logic is one of the most important daily behaviors.

state

confidence

evidence

trend

risk

history

That is why the diagnostic engine must detect fragility explicitly.

12. Layer E: Readiness composition layer
Now the big question:

How do we calculate readiness?

Readiness should be built from multiple subcomponents.

Not one flat number.

For each target exam horizon, readiness should be composed from:

Core readiness components
Coverage readiness
How much of the required content has been meaningfully touched.

Mastery readiness
How well the relevant topics are currently understood/performed.

Memory readiness
Whether the important material is stable in memory.

Condition readiness
Whether performance holds under expected exam conditions.

Expression readiness
Whether the child can answer in the required response style.

Tactical readiness
Whether the child is ready for the exam’s likely style/scope right now.

Consistency readiness
Whether performance is reliable, not noisy.

These components should form the readiness picture.

13. Readiness must be exam-relative
This is very important.

The system should not ask:

Is the child ready?

It should ask:

Ready for what?

Examples:

ready for next week’s internal test on selected topics

ready for a mock exam covering 60% of syllabus

ready for the final national exam

ready for objective section

ready for Section B explanations

ready for timed mixed paper

So readiness is always tied to:

exam horizon

subject

section or mode if relevant

topic scope

This keeps it truthful.

14. Example readiness decomposition
Let’s say the target is:

The system might produce:

mastery readiness: weak-moderate

memory readiness: fragile

condition readiness: weak under timing

expression/method readiness: moderate

tactical readiness: weak in likely school-style word problems

Overall readiness:
Not ready yet — major risk in Algebraic Expressions and timed problem solving

That is much more powerful than one score.

16. Subject readiness vs topic readiness vs exam readiness
These should be separate outputs.

Topic readiness
Can the child handle this topic for the relevant exam demand?

Subject readiness
How prepared is the child across all relevant topics in this subject?

Exam readiness
How prepared is the child for the overall target exam horizon?

These are not interchangeable.

A child may be:

ready in English grammar

fragile in Science explanations

not ready overall for the mock exam

So the engine must keep these scopes distinct.

17. Readiness gap modeling
The engine should not just emit readiness state.
It should also define the gap.

Readiness gap means:

what is missing

how much is missing

what kind of missing it is

Examples:

weak retention in high-frequency topics

poor timing performance

weak structured-answer expression

inconsistent method execution

This is incredibly useful because it tells the coach what to fix next.

21. Condition-aware diagnostics

The system should be able to say:

23. Diagnostic confidence
Just like other engines, this one must track confidence.

For any diagnostic conclusion, store:

confidence

contradiction level

Examples:

This is important so the system knows when to do deeper probing.

24. Contradiction handling
Diagnostics will sometimes conflict with earlier beliefs.

Example:

The engine should not flatten this badly.

topic may be stable in recall

but weak in explanation mode

therefore readiness for Section B is weak

This is why multi-dimensional readiness matters.

30. Operating laws
The Diagnostic and Readiness Engine should obey these laws:

Law 1
Do not reduce readiness to one number.

Law 2
Diagnosis and readiness are related but different.

Law 3
Probe as narrowly as possible while still answering the question.

Law 4
Fragility must be detected separately from obvious weakness.

Law 5
Readiness must be exam-relative.

Law 6
Condition, memory, and expression readiness matter just as much as raw topic knowledge.

Law 7
Every readiness judgment should explain what is missing and what the coach will do next.

Those are the correct laws.

3. Measure readiness by component
Coverage, mastery, memory, condition, expression, tactical fit.

4. Tie readiness to exam horizons
Internal exam, mock, final exam, and even section-level readiness.

5. Trigger the right next action
So the coach can close the correct gap efficiently.

That is the correct architecture.

29. Main end-to-end flow: upload to action
This is one of the most important product flows.

Student/parent flow
upload class test / marked script / homework

system creates bundle

system reconstructs and analyzes

if needed, user confirms ambiguous parts

Smart Review becomes available

student/parent sees weak/strong topics and why

coach updates goals and weekly plan

new missions / retests / memory work are assigned

Readiness may update

parent summary may update

This is one of the central loops of the product.

30. Main end-to-end flow: weak topic to recovery
system detects weak topic

Coach Reaction Engine classifies cause

Question/Assignment Engine produces the right mission type

weekly planner places it

daily planner adapts execution

student completes work

retest validates

Student Model updates

readiness adjusts

topic moves to fragile, stable, or remains weak

That is the core academic recovery loop.

31. Main end-to-end flow: exam horizon planning
student/parent sets exam dates

goals and exam timeline update

readiness is computed by horizon

weekly plan balances nearest and final prep

daily plan operationalizes it

evidence from real school work recalibrates the plan

readiness re-evaluates

coach changes intensities and missions as the exam approaches

This is your “parallel wheels” architecture made visible.

32. Key product behavior rules
CoachHub surfaces should obey these rules:

Rule 1
Always show the student what matters most now.

Rule 2
Always connect evidence to action.

Rule 3
Always connect action to a reason.

Rule 4
Do not expose raw system complexity unless the user needs it.

Rule 5
Student surfaces should be action-first.
Parent surfaces should be insight-first.
Admin surfaces should be control-first.

Rule 6
Every important state change should be visible somewhere meaningful.

Rule 7
The product should always feel like one connected coach, not many separate tools.

These rules matter as much as the engines.

35. Clean final summary
So the CoachHub operating surfaces should be built around three audiences.

For the student
CoachHub should feel like:

a focused daily coach

a smart reviewer of real school work

a planner that understands multiple exam horizons

a system that shows what to do and why

For the parent
CoachHub should feel like:

a clear, evidence-based academic oversight tool

a way to upload real school materials

a way to see risks, progress, readiness, and coach actions

For super admin
CoachHub should feel like:

an academic intelligence control room

a place to monitor bundle processing, interventions, readiness, and content quality

a way to keep the whole system trustworthy and high quality

And across all of them, the system should feel unified:

evidence flows into review

review flows into diagnosis

diagnosis flows into planning

planning flows into missions

missions flow into retests

retests flow back into the student model

readiness updates continuously

That is the complete CoachHub product architecture.
