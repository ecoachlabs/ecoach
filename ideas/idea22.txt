Suggested classification pipeline logic
Layer A: Rule engine
Fast deterministic heuristics.

Example:

If stem starts with "Define" -> strong signal for definition
If stem contains "Explain why" -> explanation + justification
If stem contains "Show that" -> proof / derivation
If numeric values + formula symbol + unknown requested -> worked example / application
If asks for "state two reasons" -> explanation / cause-effect
Layer B: Semantic classifier
LLM or ML classifier looks at:

stem

answer

explanation

subject

topic

options

It returns structured labels with confidence.

Layer C: Resolver
This merges rule outputs and model outputs.

Example:

rules say definition = 0.92

model says definition = 0.81

final label = definition

Or:

rules say worked_example = 0.66

model says application = 0.88

final result = both, with application primary

Layer D: Review queue
Low-confidence cases go to manual review.

Important: store primary and secondary labels
For example:

A math question may be:

primary knowledge role: worked_example

secondary knowledge role: formula_use

primary cognitive demand: application

secondary cognitive demand: recall

primary solve pattern: substitution

secondary solve pattern: rearrangement

This is much more realistic.

CREATE TABLE questions (

9. Classification pipeline
This is the flow that every question should pass through.

Stage 1 — Ingestion
Question enters from:

teacher upload

past paper import

manual entry

question generator

AI-generated content

external data import

The system stores raw data.

Stage 2 — Parsing
The parser extracts:

question stem

option structure

answer

explanation

numeric features

symbols

verbs

topic hints

diagram presence

command phrases

marks

format

This becomes a normalized parse object.

Stage 3 — Rule-based preclassification
A deterministic engine assigns early signals.

Examples:

Definition rules
starts with “define”

includes “what is meant by”

includes “meaning of”

asks for “best definition”

Explanation rules
starts with “explain”

“give reason”

“why does”

“account for”

Proof / derivation rules
“prove that”

“show that”

“derive”

“demonstrate that”

Worked example signals
has numbers

has target unknown

references known formula or variable relationship

computational structure

Diagram-reading signals
contains image/table/graph

“from the diagram”

“study the graph”

“use the table”

The rule engine should produce weighted signals, not hard final labels.

Example output:

{
  "knowledge_role": {
    "definition": 0.91,
    "key_concept": 0.48
  },
  "cognitive_demand": {
    "recall": 0.87
  },
  "solve_pattern": {
    "direct_retrieval": 0.83
  }
}
Stage 4 — Semantic model classification
Use an LLM or semantic model to classify the question more intelligently.

Inputs:

stem

options

answer

explanation

subject

topic

subtopic

marks

format

Output should be strict structured JSON, not prose.

Example:

  "knowledge_roles": [

  "cognitive_demands": [
    {"code": "application", "confidence": 0.89}

Stage 5 — Resolver
This combines rules + model + ontology constraints.

For example:

if rule confidence and model confidence strongly agree, mark high confidence

if they conflict, reduce confidence

if ontology forbids certain combinations, drop invalid labels

if a question has graph asset + graph-reading cue, boost graph_reading

This produces the final classification.

Stage 6 — Family matching
The question is compared against known families using:

semantic similarity

solve pattern similarity

concept similarity

structural pattern

numeric template similarity

explanation pattern similarity

If a match is strong enough:

assign existing family

If not:

create candidate family or flag for clustering batch

Stage 7 — Misconception mapping
Question is mapped to likely misconceptions.

Methods:

rule-based mapping from distractors

pattern mapping from solve pattern

topic-specific misconception registry

historical student error data

Stage 8 — Persistence
Write:

join-table labels

summary row

family membership

misconception links

snapshot JSON

Stage 9 — Review routing
If confidence below threshold, send to review queue.

Example thresholds:

0.88 = auto-approved

0.70 to 0.88 = usable but flagged

< 0.70 = manual review required

10. Resolver logic
You need explicit rules for how final labels are chosen.

10.1 Primary label rule
Each axis must have one primary label unless confidence is extremely low.

Example:

knowledge_roles: worked_example 0.92, application_scenario 0.61

primary = worked_example

10.2 Multi-label rule
Allow secondary labels when their confidence is close enough.

Example:

primary if within highest score

add secondary if score >= 0.60 and within 0.25 of primary

10.3 Mutual exclusion rule
Some pairs should not co-exist as primaries.

Example:

direct_retrieval and proof_chain cannot both be primary solve pattern

10.4 Compatible-pair rule
Some pairs are commonly paired.

Examples:

definition + recall + direct_retrieval

worked_example + application + substitution

proof + justification + proof_chain

explanation + comprehension + concept_based_reasoning

This improves final classification stability.

3. Product goals
Primary goals
The engine must:

classify every question across a multi-axis intelligence framework

Secondary goals

Those may sit around it, but they are not the engine’s immediate job.

5. Core design principles
Principle 1: One question, many intelligence layers
A question must not be reduced to a single question_type.

A question can simultaneously be:

centered on a key concept

testing application

solved by substitution

functioning as a bridge from concept to method

linked to a common misconception

So the model must be multi-axis.

Principle 2: Taxonomy before AI
The platform must define valid classifications first.

AI should classify into your structure.
It should not invent the structure.

Principle 3: Intelligence lives in the data layer
This must be stored in the database and exposed through APIs.

It should not be recreated feature by feature in the frontend.

Principle 4: Machine-first, reviewable, auditable
The system can classify automatically, but it must be reviewable and correctable.

Principle 5: Family logic is foundational
Recurring question families are central to transforming past questions into a superior experience.

12. Classification logic in detail
The engine should not just “guess.”
It should behave like a layered decision system.

12.1 Rule signal examples
Definition cues
“Define…”

“What is…”

“What is meant by…”

“Meaning of…”

Explanation cues
“Explain…”

“Why…”

“Give reasons…”

“Account for…”

Proof cues
“Prove that…”

“Show that…”

“Derive…”

“Demonstrate…”

Worked example cues
numeric values

target unknown

formula-like relationships

asks to calculate or solve

Graph/diagram cues
“Use the diagram”

“Study the graph”

“From the table”

question asset type is graph/table/image

Comparison cues
“Differentiate”

“Distinguish”

“Compare”

“Contrast”

Error spotting cues
“What is wrong with…”

“Identify the mistake”

“Which step is incorrect”

These become weighted features, not final decisions by themselves.

12.2 Semantic classification behavior
The semantic model should:

understand the topic context

avoid shallow keyword-only classification

use answer and explanation, not just stem

map into approved codes only

assign confidence per axis

provide a short rationale for internal logging

It must return structured JSON, not essays.

12.3 Resolver behavior
The resolver should apply logic like this:

if rules and model strongly agree, boost confidence

if rules and model disagree, reduce confidence

if taxonomy says a label combination is impossible, reject it

if graph asset exists, graph-related labels get boosted

if marks are high and the question has reasoning verbs, higher-order cognitive labels get boosted

if answer explanation contains proof language, proof-related labels get boosted

It should also enforce a primary label rule.

9. Resolver rules
This is one of the most important pieces.

The resolver should not simply average scores.
It should make educational decisions in a controlled way.

9.1 Score inputs
For each axis, the resolver receives:

rule score per label

semantic score per label

optional historical prior

optional source-based adjustment

ontology constraints

9.2 Combined score formula
A simple and strong starting formula:

combined_score =
  (rule_score * 0.40) +
  (semantic_score * 0.50) +
  (context_bonus * 0.10)
Where:

context_bonus comes from known structure, assets, marks, format, or topic cues

This can be tuned later.

9.3 Primary label rule
Each axis should assign one primary label if the top combined score is at least the minimum primary threshold.

Starting thresholds
primary threshold: 0.62

secondary threshold: 0.45

auto-approve threshold: 0.85

manual review threshold: < 0.70

9.4 Secondary label rule
A secondary label is allowed if:

score >= 0.45

and within 0.20 of the primary label

and not blocked by ontology incompatibility

9.5 Tie-breaking rule
If two labels are within 0.05 of each other:

prefer the one more central to the question’s actual task

use explanation and answer structure as tie-breakers

if still ambiguous, lower machine confidence and mark for review

9.6 Ontology compatibility rules
Examples:

Invalid or unlikely pairs
definition + proof should rarely co-exist as primary knowledge role

direct_retrieval + proof_chain cannot both be primary solve pattern

recall + proof_reasoning cannot both be primary cognitive demand

Valid common pairs
definition + recall + direct_retrieval

worked_example + application + substitution

proof + justification + proof_chain

explanation + comprehension + multi_step_reasoning

graph_diagram_reading + recognition or interpretation

9.7 Context bonuses
Examples:

graph asset present → boost graph_diagram_reading, graph_reading, table_interpretation

phrase “study the diagram” → strong boost to representational labels

phrase “show that” → boost proof, justification, proof_chain

phrase “explain why” → boost explanation, comprehension, multi_step_reasoning

numeric formula-heavy question → boost worked_example, application, substitution or rearrangement

9.8 Confidence calculation
Machine confidence should not be identical to top score.

Use a weighted function such as:

machine_confidence =
  (top_axis_average * 0.60) +
  (axis_consistency * 0.20) +
  (rule_model_agreement * 0.20)
Where:

top_axis_average = mean of primary label scores across major axes

axis_consistency = whether labels make coherent sense together

rule_model_agreement = how closely rule and semantic systems agree

10.1 Review states

10.2 Meaning of each state
pending
Classification has run but no final decision yet.

approved
Machine-approved or human-approved and accepted for use.

needs_review
Safe to store, but not trusted enough for production usage in high-stakes flows.

rejected
Classification result is invalid and must be rerun or manually fixed.

edited_and_approved
Human reviewer changed labels and approved final result.

taxonomy_gap
The reviewer found that the question cannot be adequately described using current taxonomy.

family_unresolved
Main labels may be okay, but family assignment is unclear.

archived
Question or intelligence profile is no longer active.

Flow A — New question ingestion
Step 1
A question is created.

Event:

Payload includes:

Step 2
The system emits:
question.classification.requested

Step 3
QuestionParser runs.

If parser fails:

emit question.classification.failed

mark review state needs_review

Step 4
RuleClassifier runs.

Step 5
SemanticClassifier runs.

Step 6
ClassificationResolver merges outputs.

Step 7
FamilyEngine matches or suggests family.

Step 8
MisconceptionEngine assigns likely misconceptions.

Step 9
IntelligenceRepository persists:

summary row

join rows

snapshot JSON

review state

Step 10
If confidence high:

emit question.classification.completed

If confidence low:

emit question.review.required

Flow B — Question updated
If stem, answer, options, explanation, or topic changes, the system should reclassify.

Event:
question.updated

Rules:

if only metadata unrelated to meaning changes, no reclassification

if content or topic changes, reclassify

if taxonomy version changed and question is affected, schedule reclassification batch

Flow C — Human review
Step 1
Reviewer opens queue item.

Step 2
Reviewer approves, edits, or rejects.

Step 3
System stores review record and updated intelligence.

Step 4
Event emitted:
question.review.completed

If reviewer created new family candidate:

emit question.family.unresolved or family merge event

Flow D — Taxonomy update
If taxonomy changes:

detect impacted labels

schedule batch reclassification for affected questions

preserve prior version for audit

mark snapshot as stale until refreshed

Event:
taxonomy.updated

20. Reclassification rules
Not every question update should re-run full classification.

Reclassify fully if:
stem changed

options changed

answer changed

explanation changed

subject/topic/subtopic changed

new assets added that affect interpretation

taxonomy changed in relevant area

family engine improved materially

reviewer requested reclassification

Do not reclassify fully if:
display order changed

typo fixed outside semantic meaning

source reference changed

tags unrelated to intelligence changed

23. Failure handling
The service must fail safely.

23.1 Semantic model failure
If semantic classifier times out or returns bad JSON:

fall back to rule-only classification

reduce confidence

mark needs_review

23.2 Parser failure
If parser cannot normalize question:

store failure reason

mark needs_review

do not auto-approve

23.3 Taxonomy validation failure
If model returns invalid codes:

discard invalid labels

rerun with stricter prompt if allowed

if still invalid, mark failed and send to review

23.4 Family engine uncertainty
If family similarity low:

leave family unresolved

allow question classification to remain usable

flag family review separately

This is important. Family uncertainty should not block the rest of classification.
