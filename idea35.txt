Thought for 54s
Maintain Rust for this rebuild.

For your specific situation, I would not switch the main backend to Go now. Tauri’s primary application model is Rust-native: the frontend calls Rust through Tauri commands, and Tauri positions Rust as the standard backend path in its desktop architecture. 

The real issue is not that Rust was the wrong language. The issue was that the project was not protected well enough and probably was not reconstructed into a stable, documented architecture. Rewriting the backend in Go now would add a second major risk at the same time: you would be rebuilding the product and changing the backend/runtime model. That is usually the wrong move under a deadline.

Go only becomes the better choice here in two cases. First, if you want to abandon the current Tauri-centered architecture and move to a Go-first desktop framework. Wails is that kind of model: its docs describe the desktop app as a standard Go application with a webkit frontend, with Go methods bound directly into the frontend. That is a different application shape, not a small swap. 
 Second, if you want to keep Tauri but run Go as a separate sidecar. Tauri supports that, but the sidecar setup adds packaging and deployment complexity because external binaries must be bundled and matched per target triple. 

So the practical answer is:

Keep Rust for the core desktop/backend.
Do that for:

domain models

local database and persistence

command handlers

session/runtime logic

curriculum, diagnostics, scheduling, and orchestration

all app-critical logic

That fits the stack you already had, and it keeps your rebuild aligned with how Tauri is designed to work. Tauri also keeps the Windows packaging path straightforward through its normal build flow. 

If you want to introduce Go later, use it only in a bounded way, not as a rewrite target. Good candidates would be:

a separate import/conversion utility

a reporting/export worker

a background service that is isolated from the core app

a future server component if eCoach grows beyond local desktop

My recommendation in one sentence:

Do not switch the main backend to Go; keep Rust, stabilize the architecture, and rebuild in disciplined slices.

The stronger version of the decision is this:

If you are staying on Tauri, stay on Rust

If you truly want Go-first, then that is really a framework migration decision, not just a backend swap
