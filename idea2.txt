4. Journey should define mastery carefully

“Mastery” can mean different things:

Version A — Exam readiness mastery

Version B — Topic mastery

Version C — Speed mastery

4. Misconception Engine
This is one of the most important parts of Journey.

Wrong answers should not be lazy distractors.

They should be deliberately engineered to represent plausible wrong thinking.

So when the learner chooses a wrong answer, the system learns something useful.

For each concept, you should build a misconception library.

For example in fractions:

Question:
What is 1/2 + 1/3?

Options:

2/5

5/6

1/5

2/6

These wrong answers are not random:

2/5 = adds numerator to numerator and denominator to denominator

1/5 = adds denominator only / confusion

2/6 = partial structure confusion or same-denominator simplification mistake

So if the learner picks 2/5, that reveals a specific misconception:

“Learner treats fractions like whole numbers and adds components directly.”

That means the wrong answer is a diagnosis.

This engine decides whether evidence is weak, moderate, or strong.

7. Linked questions: what that should really mean
You mentioned that some questions are linked and the engine is thinking about coverage. Correct.

Linked questions should not just mean “same topic.”

There are several link types.

A. Prerequisite link
Question B depends on the concept behind Question A.

Example:

simplify fractions

then solve ratio problem

then solve a proportional reasoning question

If the learner fails later, Journey can trace backwards.

B. Misconception link
Different questions test the same misconception in different disguises.

Example:
A learner chooses wrong answers that suggest:

sign error in algebra

confusion between denominator and numerator

misunderstanding of probability as ratio

The engine links those to a misconception family.

C. Confirmation link
A second or third question tests whether a previous success was real.

Example:
The learner gets:
3x + 5 = 17
right.

Confirmation link then gives:
5 + 3x = 17
or a word problem version, or a negative-number version.

Now the engine knows whether they understand or merely recognized a pattern.

D. Transfer link
Same concept, different surface form.

Example:

direct arithmetic

word problem

diagram

real-life context

abstract symbolic form

E. Escalation link
Questions increase in complexity to see how far competence extends.

F. Coverage link
Questions touch neighboring concept regions so the engine knows how broad the learner’s ability is.

So the learner is not just answering isolated questions. They are moving through an evidence web.

Journey Core Engine Specification
1. What the Core Engine is
The Journey Core Engine is the master system that manages a learner’s path through a subject from current state to target readiness.

It is responsible for:

3. The fundamental design principles
The engine should obey these rules:

Rule 1 — The subject is a graph, not a list

Journey Time & Readjustment Logic Specification
1. What this layer is
The Time & Readjustment layer is the supervisory logic that continuously recalculates the smartest route from the learner’s current state to their target exam goal.

Its job is not to teach directly. Its job is to govern the pace, depth, intensity, and proof burden of the journey.

It answers questions like:

Are we on track?

How much of the route is still left?

How much proof do we still need?

Can this learner still reach the target at the current pace?

Should we deepen or compress?

Should we add more mini-mocks?

Should we reduce low-value repetition?

Should we switch to high-yield mode?

Should we rescue the route?

This is the strategic layer above the core engine.

2. The central promise
This layer should make Journey feel like:

“The system is not just teaching me. It is constantly recalculating the smartest way to get me ready in time.”

That is what makes it powerful.

3. The main design principle
The journey should not be fixed.

It should be continuously shaped by four realities:

A. Goal reality
What does “ready” mean for this learner?

B. Time reality
How much usable time is left?

C. Progress reality
How fast is the learner truly improving?

D. Risk reality
What still threatens success?

The job of this layer is to keep these four realities aligned.

4. What this engine must track
To make good decisions, it needs a live state model.

4.1 Time state
exam date

today’s date

days remaining

realistic study days remaining

average available minutes per day

expected future availability

missed-day risk

buffer days remaining

4.2 Goal state
target exam

target score / grade / readiness threshold

minimum acceptable outcome

preferred outcome

deep mastery goal vs exam readiness goal

4.3 Progress state
route completion

mastery completion

retention stability

concept velocity

station velocity

mini-mock trend

challenge performance trend

readiness trend

current route mode effectiveness

4.4 Risk state
fragile prerequisite clusters

high-exam-weight weak zones

poor formula recall

unstable transfer performance

poor timed performance

inconsistency in study pattern

repeated misconception resurfacing

overconfidence / underconfidence gap

5. The three key planning quantities
This layer needs three central computed quantities.

5.1 Remaining Work
How much route work is still unfinished?

This includes:

unvisited stations

uncleared stations

fragile stations requiring confirmation

overdue retention checks

planned mini-mocks

remaining readiness gates

This should not be measured only as “number of topics.”
It should be measured as weighted effort.

Weighted remaining work should account for:
difficulty of node

exam weight

learner weakness severity

proof still needed

whether retention is already strong or still fragile

5.2 Available Capacity
How much realistic study capacity remains before the exam?

This includes:

planned study days

expected daily study duration

actual consistency history

fatigue allowance

missed-day expectation

revision buffer

So the system should not use fantasy time.
It should use probable usable time.

5.3 Goal Gap
How far is the learner from target readiness?

This should combine:

mastery gap

retention gap

speed gap

formula / definition recall gap

exam integration gap

mock score gap

This tells the system how far it still needs to push.

6. The main job of this layer
The engine continuously tries to maintain this balance:

Available Capacity >= Required Work to close Goal Gap

If that inequality becomes weak or false, Journey must adapt.

That is the whole purpose of the layer.

7. The strategic outputs of this layer
This engine should produce decisions like:

continue as planned

deepen the route

slightly compress the route

aggressively compress the route

intensify formula/definition recall

add more mini-mocks

reduce low-value repetition

shift to high-yield focus

prioritize weak high-weight concepts

switch to rescue mode

recommend goal adjustment

These are strategic outputs, not just analytics.

9. The main calculations this layer needs
Not rigid final formulas, but clean planning signals.

9.1 Readiness Score
This is the learner’s current estimated exam readiness.

It should combine:

mastery coverage

retention confidence

transfer ability

timed performance

mini-mock performance

formula / definition recall

consistency trend

Example conceptual weighting:

25% mastery coverage

15% retention stability

15% timed performance

15% mock integration

10% formula/definition recall

10% transfer performance

10% consistency and recency

This can vary by subject.

9.2 Work Remaining Score
Weighted estimate of remaining route effort.

For each unfinished or unstable node, compute something like:

node_remaining_work = exam_weight × weakness_severity × proof_required × dependency_importance

Then sum across the route.

This makes remaining work realistic.

9.3 Capacity Score
Estimated usable effort before the exam.

Conceptually:
capacity = remaining_days × average_usable_session_time × consistency_factor × fatigue_factor

Where:

consistency factor reduces fantasy capacity if learner often misses sessions

fatigue factor prevents overpacking

This gives probable capacity, not ideal capacity.

9.4 Velocity Score
How quickly this learner is improving.

This should reflect:

station clearance rate

mastery increase rate

misconception reduction rate

readiness increase rate

mock improvement rate

Journey Assessment Modes Specification
1. What this layer is

This layer answers questions like:

Should I test recall or application right now?

Should this be a mini-mock or a repair drill?

Does the learner need pressure or stability first?

Should this concept be tested in direct form, transfer form, or mixed form?

Is this a formula check, a misconception probe, or a station challenge?

What kind of testing mix is right for this learner in this subject at this moment?

This layer sits between the route logic and the actual question flow.

2. The main design rule
Assessment modes should not be treated as content wrappers.

Each mode must be defined by:

its evidence purpose

its emotional effect

its place in the route

its subject adaptation

its learner adaptation

That means Journey should not say:

“Let’s give a quiz.”

It should say:

“The learner now needs a short transfer test because direct performance looks good but flexible use is still unproven.”

That is the right level of intelligence.

Journey UX Surfaces Specification
1. What this layer is
The UX Surfaces layer is the set of interfaces through which the learner experiences Journey.

It should expose the engine’s intelligence without overwhelming the learner.

The main rule is:

The user should always know three things:

where they are

why they are here

That clarity is essential.

2. The core UX principles
Every Journey surface should obey these principles:

Principle 1 — Resume, don’t restart
The learner should always feel that the subject remembers them.

Principle 2 — Show movement, not just numbers
Progress should feel territorial, structural, and alive.

Principle 3 — Explain intent selectively
The system should sometimes reveal why it is asking something.

Principle 4 — Reduce friction into the next action
The next best move should be obvious.

Principle 5 — Make mastery visible at multiple scales
The learner should see local, station, topic, and full-route progress.

Principle 6 — Keep emotional tone stable
Even when the learner is behind, the UI should feel recoverable, not punishing.

Principle 7 — Subject flavor must be preserved
Math should not look and feel identical to Language.

Journey Data Model & State Machine Specification
1. What this layer is
The Data Model & State Machine layer is the canonical representation of Journey in system terms.

It has two jobs:

Data model job
Define the durable objects and fields the product needs.

State machine job
Define how those objects move from one valid state to another.

This layer should be treated as the single source of truth for Journey behavior.

2. Core design principles
Principle 1 — Persist state at subject level
Journey is saved under each subject. A learner can be in different route states across subjects.

Principle 2 — Separate authored structure from learner state
The subject blueprint is authored once; learner state evolves against it.

Principle 3 — Preserve history, not just current position
Journey must remember where the learner has been, not only where they are now.

Principle 4 — Use state labels that mean something
State should reflect pedagogical truth: fragile, retained, at-risk, needs-reactivation, etc.

Principle 5 — Events drive state change
Answers, time gaps, mini-mock outcomes, missed days, and re-entry probes must all create explicit events.

Principle 6 — Transitions should be deterministic
Given the same inputs and thresholds, the engine should make the same transition.

2. Architectural principles
This system should obey these principles:

Principle 1 — Separate authored intelligence from learner state
Subject blueprints, question families, and misconception libraries should not live in the same logic path as mutable learner progress.

Principle 2 — Keep orchestration separate from scoring logic
The service that decides “what happens next” should not be the same service that renders UI or stores raw responses.

Principle 3 — Prefer event-driven updates internally
Journey is a stateful system. Answers, time gaps, re-entry, mini-mocks, and plan shifts all behave naturally as events.

Principle 4 — The frontend should consume high-level Journey views, not assemble them from raw objects
Frontend complexity will explode if it has to reconstruct route logic itself.

Principle 5 — Question generation must be controlled and validated
Never allow raw generation to directly drive learner assessment without a validation layer.

Principle 6 — Resume and sync must be first-class
A learner should be able to stop and continue without corrupting route state.

Journey Spec 8: Question Generation & Validation Framework
1. What this layer is
The Question Generation & Validation Framework is the system that decides:

what kind of question to ask

how to produce it safely

how to keep it fresh

how to ensure it tests the intended skill

how to make wrong answers meaningful

how to validate that the item is correct and usable before the learner sees it

Its job is not just to create questions.
Its job is to create evidence instruments.

That means every question must preserve:

concept target

skill target

mastery level fit

testing reason

misconception value

difficulty intent

freshness quality

assessment reliability

That is the real purpose of this framework.

2. Core design principles
Principle 1 — Generation must preserve evaluative intent
A new question is only useful if it still tests the same underlying thing.

Principle 2 — Retrieval and generation must coexist
Some questions should be fixed and curated. Others should be generated from families.

Principle 3 — Distractors must be deliberate
Wrong answers should represent plausible thinking paths, not random noise.

Principle 4 — Freshness must be controlled
Newness should come from meaningful variation, not chaotic randomness.

Principle 5 — Validation is mandatory
No generated question should be shown until it passes structural and pedagogical checks.

Principle 6 — Offline determinism matters
Given the same family, parameters, and rules, the generator should behave predictably and testably.

Principle 7 — Different modes need different generation styles
A repair item is not built the same way as a mini-mock or readiness gate item.

3. The generation architecture
A strong offline framework should use a hybrid question architecture with three layers.

3.1 Layer A — Fixed Curated Items
These are hand-authored questions.

Use for:

critical checkpoints

edge-case misconceptions

final readiness gates

mini-mock anchors

challenge boss items

especially tricky transfer items

These provide high trust.

3.2 Layer B — Parametric Template Generation
This is the core freshness engine.

It generates questions by varying:

values

structures

contexts

wordings

representation forms

distractors

while preserving the same target skill and testing reason.

This should be the main generation method for offline Journey.

3.3 Layer C — Structural Variation Engine
This sits on top of families and changes the surface while keeping the same evaluative purpose.

Examples:

symbolic → word problem

direct → reversed form

single-step → missing-step

solve → find the mistake

choose the answer → choose the correct method

This makes the system feel smarter and less repetitive.

4. The central object: Question Family
The most important authored object in this framework is the Question Family.

A Question Family is not a single question.
It is a formalized pattern for testing one precise thing in many valid ways.

4.1 What a Question Family must define
Each family should contain:

family identity

concept target

skill target

supported mastery levels

supported testing reasons

valid generation parameters

surface form options

difficulty controls

answer derivation rules

distractor recipes

misconception mappings

freshness rules

validation rules

explanation scaffolds if needed

4.2 Example family
Family name:
Use LCM to create common denominators for unlike fractions

This family might support:

direct multiple choice

short answer

equivalent fraction rewriting

word-based setup

error diagnosis format

timed drill form

All of those can still test the same core skill.

That is what a family is for.

5. The internal anatomy of a Question Family
Each family should be defined using these sections.

5.1 Family Metadata
Defines:

question_family_id

subject_id

concept_id

skill_id

family_name

description

author_version

5.2 Evaluative Intent Profile
Defines what the family is truly testing.

Fields:

primary_skill_target

secondary_skill_targets

supported_testing_reasons

supported_mastery_levels

supported_modes

evidence_strength_profile

This is what keeps generation aligned to pedagogy.

5.3 Parameter Space
Defines the valid values and ranges the generator can use.

Examples:

denominator range: 2–12

allow coprime only: true/false

max number of steps: 2

allow improper fractions: true/false

include mixed numbers: true/false

These parameters are the core of deterministic variation.

5.4 Surface Forms
Defines how the same family can appear.

Examples:

direct symbolic

direct verbal

multiple choice

short answer

matching

fill-in-step

identify the error

compare two solutions

mini-case or context scenario

This gives freshness without losing control.

5.5 Difficulty Controls
Defines the knobs that raise or lower complexity.

Examples:

size of numbers

number of steps

presence of traps

closeness of distractors

amount of disguise

context complexity

required integration with prior concepts

Difficulty should be produced by structured dimensions, not by vague labels only.

5.6 Answer Logic
Defines how the correct answer is computed.

This must be explicit and deterministic.

Examples:

compute LCM

convert both fractions to equivalent forms

derive correct sum

derive final simplified answer

Every generated item must be backed by local answer logic.

5.7 Distractor Recipes
Defines how wrong answers are built.

Each distractor should come from:

a known misconception

a known step error

a known shortcut misuse

a plausible confusion pattern

This is one of the most important parts of the framework.

5.8 Validation Rules
Each family must define what validity means.

Examples:

exactly one correct answer

distractors must be unique

distractors must not accidentally equal the correct answer

generated values must stay within intended difficulty band

generated wording must stay grammatically valid

no impossible or ambiguous case allowed

5.9 Freshness Rules
Defines how repetition is controlled.

Examples:

do not reuse same parameter tuple too soon

vary surface form after N uses

avoid identical distractor ordering

rotate context patterns

avoid repeating same trap family consecutively unless in confirmation mode

6. The types of generation supported
An offline-safe system should support these generation types.

6.1 Numeric Parameter Variation
Change the values while keeping structure constant.

Example:

1/2 + 1/3

3/4 + 1/6

5/8 + 3/10

This is the simplest variation type.

6.2 Structural Variation
Change the shape of the task.

Examples:

solve directly

find common denominator only

rewrite fractions first

choose which denominator is least common

identify wrong student step

This helps test deeper understanding.

6.3 Representation Variation
Change how the idea is presented.

Examples:

symbolic form

verbal form

contextual word problem

diagram or table form where relevant

This is essential for transfer.

6.4 Distractor Variation
Rotate which misconception is foregrounded.

Examples:

one version targets denominator addition confusion

another targets numerator scaling confusion

another targets using a common multiple that is not least

another targets correct LCM but wrong equivalent fraction transformation

This keeps evidence quality high.

6.5 Sequence Variation
Change order within a set.

This is useful for:

mini-mocks

challenge modes

confirmation sequences

mixed practice

7. The role of testing reason in question generation
This is crucial.

The same family should generate differently depending on why the question is being asked.

7.1 Discovery generation
Should produce:

broad, not overly trick-heavy items

level-locating items

enough variation to detect boundary

medium evidence strength

7.2 Repair generation
Should produce:

narrow, focused items

misconception-targeted distractors

progressive scaffolding if needed

small-variance sets around one weakness

7.3 Confirmation generation
Should produce:

linked but not identical items

slightly shifted surface

moderate evidence strength

stability-check form

7.4 Transfer generation
Should produce:

different surface form

disguised concept usage

context shift

integration with nearby ideas

7.5 Retention generation
Should produce:

older concepts

surprise placement

not always in same form as before

enough recognizability to test memory fairly

7.6 Pressure generation
Should produce:

clean, fast-answerable items

time-sensitive versions

limited ambiguity

high fluency value

7.7 Mini-mock generation
Should produce:

balanced set composition

coverage-aware selection

stable difficulty distribution

realistic exam rhythm

7.8 Challenge generation
Should produce:

sharper objective

distinctive framing

higher emotional payoff

tighter rule set

So question generation must always receive testing reason as a first-class input.

8. Distractor design framework
This is a major moat.

Wrong answers should not be filler.
They should be cognitive probes.

8.1 Distractor types
I would define at least these types:

A. Misconception distractor
Directly reflects a known wrong mental model.

B. Procedure-slip distractor
Represents a common step error.

C. Overgeneralization distractor
Represents a learner applying a correct rule in the wrong place.

D. Careless-trap distractor
Represents fast but shallow processing.

E. Partial-correctness distractor
Represents getting some steps right but not finishing correctly.

F. Context-misread distractor
Useful in word problems and science/language contexts.

These categories help standardize authoring and analytics.

8.2 Every distractor should carry metadata
Recommended fields:

distractor_id

distractor_type

misconception_id

reasoning_path_summary

confidence_of_signal

difficulty_appropriateness

allowed_mastery_levels

This lets the engine learn from wrong answers properly.

8.3 Distractor generation rules
A distractor should be:

plausible

distinct

not absurd

not accidentally correct

aligned to one or more target misconceptions

appropriately deceptive for the learner’s level

Bad distractors make the whole system less intelligent.

9. Freshness control framework
The product promise includes freshness, but freshness must be engineered.

9.1 Freshness dimensions
Freshness can come from:

new values

new wording

new ordering

new representation

new distractor profile

new context shell

new linked progression order

Not every question must be “completely novel” to feel fresh.

9.2 Anti-repetition memory
The generator should track recent exposure by:

family

parameter pattern

surface form

distractor profile

context type

exact stem shape

This history should feed generation.

9.3 Repetition rules
Examples:

do not reuse same exact parameter tuple within N recent items

rotate surface form after repeated direct use

avoid same misconception-targeted distractor twice in immediate sequence unless in repair mode

allow more structural similarity in confirmation mode

allow less similarity in normal progression mode

This is how repetition becomes controlled, not accidental.

10. Difficulty modeling
Difficulty must be explicit.

10.1 Difficulty should be multidimensional
Do not store only “easy / medium / hard.”

Store dimensions like:

computational complexity

conceptual complexity

disguise level

integration depth

time burden

trap closeness

language complexity

That gives much better control.

10.2 Mastery-level fit
Difficulty must map to mastery ladder levels.

Example:

Novice: low disguise, low step count, low trap density

Foundation: direct procedure, small variation

Developing: standard problems, some trap risk

Proficient: broader variation and moderate disguise

Advanced: transfer, integration, close distractors

Mastery: pressure, mixed form, robust transfer

This makes generation progression-aware.

11. The validation pipeline
This is non-negotiable.

Every generated or selected question should pass validation before use.

11.1 Validation layers
A strong validation pipeline should include:

Layer 1 — Structural validation
Checks:

required fields present

parameters within valid range

answer exists

distractor count valid

Layer 2 — Mathematical / logical correctness
Checks:

correct answer computed properly

distractors not accidentally correct

internal consistency of item

Layer 3 — Pedagogical validation
Checks:

item really targets the intended skill

difficulty fits intended band

distractors reflect intended misconceptions

testing reason compatibility is valid

Layer 4 — Freshness validation
Checks:

not too similar to recent items

not overused pattern

suitable for current session mix

Layer 5 — Language/render validation
Checks:

wording is readable

formatting is valid

symbols render correctly

no broken UI content

Only then should the question be exposed.

11.2 Validation outcomes
Each item should end in one of these states:

valid

valid_with_warning

rejected

quarantined_for_review

For v1, you may only surface valid items to learners.

12. Retrieved vs generated item strategy
The engine should not always generate.

A good rule system is:

Prefer curated retrieval when:
item is for final readiness gate

item is a high-stakes mini-mock anchor

item tests an edge-case misconception

item is one of a few gold-standard station checks

Prefer generation when:
freshness demand is high

the concept family is parameter-friendly

the learner needs many practice probes

repair loops need many near-equivalent variants

direct procedure families are well-defined

Prefer hybrid when:
a curated structure can be varied safely

challenge shells need controlled freshness

mini-mocks need a mix of anchor and fresh items

This gives both trust and flexibility.

13. Question selection policy
Before generating or retrieving, the engine should decide what kind of item is needed next.

Inputs should include:

subject

concept

skill

mastery level

testing reason

learner history

active misconception risk

recent item history

current session mode

deadline pressure

The selection policy then outputs:

family id

retrieve vs generate

surface form

difficulty band

distractor emphasis

timer profile

explanation style if needed

This is the bridge between route logic and actual item delivery.

14. Mode-specific item composition
Different assessment modes require different item mixes.

14.1 Repair mode composition
high family consistency

misconception-targeted distractors

lower context noise

progressive scaffolding if needed

14.2 Confirmation mode composition
linked but not identical items

surface shifts

moderate difficulty

stronger anti-cue design

14.3 Mini-mock composition
controlled spread across concepts

stable difficulty gradient

realistic exam proportions

balance of direct and applied items

14.4 Challenge composition
tight theme

memorable framing

more distinct progression

clear victory meaning

14.5 Pressure mode composition
clean stems

timer-ready answer paths

low ambiguity, high speed value

So the framework must support composition policies, not only single-item generation.

15. Explanation scaffolds
This is optional for some items, but very useful.

Each family can define explanation scaffolds like:

correct reasoning template

misconception explanation template

hint template

feedback summary template

This allows Journey to say things like:

“You found the LCM correctly, but scaled only the denominator.”

“This question checks whether you still remember how to create equivalent fractions.”

That makes post-answer feedback stronger and more consistent offline.

16. Family authoring rules
To keep quality high, each question family should be authored with these minimum rules:

Must define
one primary skill target

at least one supported testing reason

at least one difficulty profile

answer computation logic

at least two distractor recipes if multiple choice

validation rules

freshness rules

Should define
surface form variants

explanation scaffolds

transfer-friendly forms where possible

mastery-level mapping

Should not allow
vague skill targeting

distractors without meaning

parameter ranges that create invalid edge cases

difficulty labels without real knobs

This gives consistency across families.

17. Quality assurance for question families
You need local QA before shipping a family.

17.1 Family-level QA
Check:

does the family really test the intended skill?

are parameters safe?

do distractors map to real misconceptions?

can the family generate enough variety?

does difficulty increase meaningfully across levels?

17.2 Generated sample QA
Generate many samples from the family and inspect:

correctness

clarity

repetition risk

difficulty spread

distractor plausibility

17.3 Simulation QA
Run simulated learners through the family:

novice-like error profiles

proficient-like profiles

misconception-specific profiles

See whether the family behaves as expected.

This is especially important offline where the system depends on local content quality.

18. Mini-mock and challenge item rules
These should not be treated like ordinary practice items.

18.1 Mini-mock item rules
Mini-mock items should be:

balanced across concepts

difficulty-calibrated

less repetitive

more exam-realistic

stable in scoring meaning

Use more curated anchors here.

18.2 Challenge item rules
Challenge items should be:

purposefully themed

emotionally distinct

more memorable

tightly aligned to one skill or trial objective

Use generation selectively and carefully here.

2. The core design principles
Principle 1 — Packs are the unit of subject delivery
Do not hardcode subject logic into the app. Ship subject packs.

Principle 2 — Schemas must exist before authoring tools
The content contract comes first. UI tools can come later.

Principle 3 — Packs must be machine-validatable
Broken references, missing misconception IDs, invalid question families, and route gaps must be caught automatically.

Principle 4 — Packs must be versioned
Curriculum structure will evolve. The app must know what version of a pack the learner is on.

Principle 5 — Pack upgrades must not destroy learner progress
Migration rules are required.

Principle 6 — Authoring should be modular
Each content concern should live in its own file or section.

3. The content pack as a product unit
A content pack should represent a bounded educational runtime domain.

23. Versioning model
This is mandatory.

Each pack must have:

pack_version

schema_version

strategy_version

migration declarations

Recommended pack version style:

semantic versioning: 1.0.0, 1.1.0, 2.0.0

What version changes mean
Patch
Typo fixes, label changes, non-structural corrections

Minor
New families, new challenges, new stations that do not break learner mapping

Major
Concept graph changes, station changes, ladder changes, route logic changes requiring migration

24. Migration model
A learner’s state must survive pack upgrades.

You need migration rules for:
concept split

concept rename

station rename

station split/merge

new ladder threshold

retired misconception ids

replaced question family ids

Example migration file
migrations/migration_1_0_0_to_1_1_0.json

This might say:

map old concept fractions_lcd to new concept fractions_common_denominators

rename station lcd_gate to common_denominator_crossing

carry node progress as-is

mark one new station as available but unattempted

That keeps learner progress intact.

25. Recommended migration strategies
Use these migration types:

Direct rename
Old id becomes new id.

Split
One old concept becomes multiple new ones.
Strategy:

distribute prior confidence conservatively

mark as partially retained / needs confirmation

Merge
Multiple old concepts combine into one.
Strategy:

use strongest compatible prior state with confirmation required

Retire
Old item removed.
Strategy:

archive historical reference but remove active route usage

New addition
New station or family added.
Strategy:

mark unlocked or locked depending on route rules

This matters a lot as Journey evolves.

26. Offline storage model for packs
The desktop app should separate:

Static content
Stored as local pack files:

JSON pack bundle

versioned

checksummed

Learner state
Stored in local DB:

current journey state

node states

station states

event history

archive

This is important because content packs and learner data should not be mixed.

27. Pack installation and updates
Since offline is mandatory, updates should be optional and local.

Possible models:

pack bundled with app install

additional packs imported manually from file

app update replaces or adds packs

local upgrade process migrates learner state

The app should support:

install pack

validate pack

activate pack

migrate learner state

rollback if migration fails

That makes the system safe.

28. Compiler / build tooling for packs
You should build a local “pack compiler” tool even before a fancy authoring UI.

Its job:

read author files

normalize ids

enforce schema

run graph checks

run family checks

compile runtime JSON

emit validation report

emit preview summaries

This will save enormous pain.

29. Preview and simulation tooling
This is another high-value tool.

Before shipping a pack, authors should be able to:

preview the concept graph

preview station map

inspect mastery ladders

generate sample questions from each family

preview distractors

simulate learner flows

simulate progression / failure / reactivation

This can be crude at first, but it is extremely useful.

30. Minimum viable content authoring setup for v1
For the first build, I would keep it practical.

Use:
JSON or YAML files

schema validators

pack compiler

sample generator

simple preview scripts

Do not require yet:
full visual authoring studio

live drag-and-drop concept graph editor

advanced collaborative authoring platform

Those can come later.

For v1, correctness beats polish.
