## System Purpose

- Explicit in idea1: build an exam-performance operating system, not a generic question bank.
- Explicit in idea1: for BECE/Ghana first, diagnose exactly why marks are being lost, close gaps faster, simulate the exam, and coach students toward readiness.
- Strong inference from idea1: the real product is a hidden intelligence layer behind a simple tests-first UX.
- Explicit in idea2: Journey should take a learner from current level to exam readiness within a chosen deadline by continuously recalculating the smartest path based on evidence, time, and subject strategy.
- Explicit in idea3: the application should operate as a local academic intelligence hub for learners, parents, and admins on a shared device, turning academic data into clear action and role-appropriate insight.
- Explicit in idea4: the system should rebuild weak or disengaged learners through root-cause diagnosis, controlled daily micro-gains, and confidence-preserving transformation rather than generic revision.
- Explicit in idea5: the system should also support already-strong learners with a syllabus-faithful elite-performance environment that sharpens precision, speed, depth, trap resistance, consistency, and pressure handling.
- Explicit in idea6: the system should continuously model what a student does not know, is weak in, is forgetting, or is starting to lose, then prioritize and solidify those gaps until the active gap shrinks.
- Explicit in idea7: the system should separately track what the learner already learned but may be forgetting, then detect, rescue, reconnect, and re-confirm that memory before performance decays.
- Explicit in idea9: the product needs a hidden question engine that generates many cognitive question types from structured concept knowledge, learner state, and evaluative intent.
- Explicit in idea10: the product may include game modes where academic answers directly control game-state privileges, pressure, and recovery rather than simply sitting beside a game as cosmetic trivia.
- Explicit in idea11: the product should include a Wrong Answer Intelligence System / Academic Analyst that reconstructs why a student chose a wrong answer, explains every option, detects patterns across history, and prescribes targeted recovery.
- Explicit in idea12: the product should also operate as a premium academic command center for parents who are buying certainty, oversight, diagnosis, intervention, and proof rather than ordinary software access.
- Explicit in idea13: the system should transform past papers from a static archive into an Exam Pattern / Past Paper Intelligence Engine that explains recurrence, mutation, co-appearance, inverse appearance, replacement, story, and student-specific weakness inside historical exam behavior.
- Explicit in idea14: the system should let students prepare for a specific upcoming assessment by modeling test type, likely scope, urgency, desired preparation style, and current scoped readiness, then composing the right assessment experience for that reality.
- Explicit in idea15: the application should become a full learning operating system that closes the loop from teaching to practice, diagnosis, repair, retention, planning, re-testing, mastery, and daily return.
- Explicit in idea16: the Library should become the app's active knowledge hub, not just a saved-questions drawer, by storing, organizing, teaching, reconnecting, recommending, preparing, and tracking academic growth.

## Business Rules

- Never predict exact future exam questions; predict exam blueprints/probabilities with uncertainty.
- Mocks compile from question families first, then render/select instances.
- Every generated question must pass hard quality gates: single correctness, clarity, misconception integrity, novelty, coverage contribution, difficulty fit, traceability.
- Every wrong answer should map to a misconception and a repair path.
- Coverage must be multidimensional and not count guessed/unstable performance as true completion.
- Frontend must never own truth for scoring, readiness, diagnosis, or mock generation.
- Journey rule from idea2: every question must have intent; if the system cannot explain what it is learning from a question, it should not ask it.
- Historical conquest is not current truth; past mastery must be rechecked via retention, transfer, and pressure evidence.
- Advancement requires explicit proof thresholds, variation handling, misconception suppression, confirmations, and sometimes timing/retention proof.
- Route compression must be strategic, not random: cut low-impact effort while preserving the highest-likelihood path to exam success.
- Offline v1 rule from idea2: deterministic authored structure and template-based variation first; no cloud AI dependency for core assessment truth.
- Idea3 rule: major learner/parent effort should be justified; every major recommendation should be explained; every important result should be interpreted.
- Role separation rule: learner needs motivation, parent needs understanding, admin needs control; these should not be treated as one UI with hidden tabs.
- Topic action rule: the same topic requires different backend flows depending on whether the learner wants instruction, repair, revision, or elite challenge.
- Beat Yesterday rule from idea4: do not aggressively push volume, correctness, and pace all at once; choose a primary growth axis and protect the others.
- Speed-pressure rule from idea4: only increase time pressure when accuracy is stable enough; speed gains that destroy correctness are invalid.
- Session-close rule from idea4: weak or disengaged learners should finish sessions with achievable wins to preserve return behavior.
- Reward rule from idea4: reward meaningful growth, completion, and consistency rather than simple logins or empty activity.
- False-growth rule from idea4: increases in attempt volume that come with guess spikes or sharp accuracy collapse should be treated as unhealthy growth and rebalanced.
- Elite rule from idea5: stay strictly within syllabus boundaries; challenge must come from execution standards, not off-syllabus content.
- Fairness rule from idea5: avoid fake difficulty, ambiguity, or punishment masquerading as challenge.
- Stability rule from idea5: do not promote or downgrade elite standing from one hot or bad session; use rolling windows and thresholds.
- Question freshness rule from idea5: avoid exact reuse too quickly, limit repeated trap patterns, and maintain purposeful session composition.
- Knowledge-gap integrity rule from idea6: the gap should only meaningfully shrink when the student demonstrates correctness, repeat correctness, variation success, retention, usable speed, and reduced misconception load.
- Real-time defense rule from idea6: falling-behind areas should rise in urgency immediately enough to trigger stabilization before weakness spreads downstream.
- Memory confirmation rule from idea7: a concept is not "in memory" because of one correct answer; it needs time-separated, non-identical, usable retrieval evidence that satisfies the subject-specific proof template.
- Mode-boundary rule from idea7: if failure is driven by misunderstanding rather than forgetting, Memory Mode should hand off to a teaching/weakness-repair mode instead of pretending memory rescue will solve it.
- Question-engine rule from idea9: a question is not just content; it is an instrument for extracting specific evidence about the learner.
- Dual-typing rule from idea9: store both answer format and cognitive type because the same visible format can test different mental actions.
- Validation rule from idea9: every generated item needs curriculum fit, answerability, diagnostic integrity, and family-specific validation before delivery.
- Educational-game rule from idea10: the learning answer must drive the game mechanic itself; quiz content glued on top of unrelated gameplay is considered weak.
- Anti-frustration rule from idea10: wrong answers should create pressure and reduced control, but the system needs mercy/recovery paths so weak learners are not punished twice into collapse spirals.
- Wrong-answer rule from idea11: every distractor should exist as a diagnosis tool with explicit intent, not as filler.
- Review-depth rule from idea11: not every wrong answer needs equal forensic depth; the system should choose light, standard, or deep review based on severity, recurrence, and value.
- Premium communication rule from idea12: interpret before visualizing; parent-facing outputs should explain current state, why it is true, what is being done, and what to expect next.
- Messaging-discipline rule from idea12: send immediately only for high-severity risks, material strategy shifts, and escalations; send routine briefs on schedule; avoid noisy low-value progress spam.
- Intervention-governance rule from idea12: every intervention must carry objective, status, review date, expected outcome, and explicit close/continue/redesign/escalate logic.
- Premium-surface rule from idea12: parent home should show only the highest-value risks and changes, not raw analytics exhaust.
- Past-paper intelligence rule from idea13: the question is not the true unit; intelligence should be organized around families, cognitive moves, misconception patterns, examiner intent, and relationship signals.
- Relationship-confidence rule from idea13: inverse, replacement, and evolution claims need support thresholds and confidence, not one-off coincidences.
- Story rule from idea13: every historical question should be interpretable as part of a larger paper/family/exam story, not only as a raw archived item.
- Customised-testing rule from idea14: likely-question logic must remain high-probability preparation, not certainty claims or guaranteed prediction.
- Simulation-integrity rule from idea14: full simulation modes should remain strict and minimally adaptive so realism is preserved.
- Scoped-readiness rule from idea14: readiness and weakness should be interpreted inside the chosen test scope, not only from global subject performance.
- Noise-avoidance rule from idea15: do not add surfaces or alerts that do not create a clear academic reason to return; every surfaced item should have urgency, actionability, and visible payoff.
- Home/mission rule from idea15: show at most a few high-value actions, each with why-now, time estimate, and likely benefit; avoid overwhelming dashboard sprawl.
- Retention-integrity rule from idea15: memory pressure, weakness closure, momentum, and readiness signals should be academically meaningful, not empty gamification.
- Library-action rule from idea16: the library must be operational and academically actionable, not passive storage; every surfaced item should lead to teach/test/revise/connect actions.
- Relationship rule from idea16: library items should not be isolated; the value comes from graph-like cross-linking between concepts, questions, mistakes, memory, and exam patterns.
